# 导航卡死问题分析与修复

## 问题描述
系统在切换导航时会出现卡死现象，特别是在刚登录的时候。

## 问题分析

### 1. AuthContext 中的竞态条件 ⚠️

**问题位置**: `src/contexts/AuthContext.tsx`

**问题原因**:
- `getUser()` 和 `onAuthStateChange` 都会设置 `loading` 状态
- `onAuthStateChange` 可能在 `getUser()` 完成前触发，导致 `loading` 被提前设置为 `false`
- 登录后，`signIn` 设置 `loading`，但 `onAuthStateChange` 也会触发，导致状态不一致
- 这会导致路由守卫在认证状态未完全确定时就进行判断，造成导航卡死

**修复方案**:
- 使用 `useRef` 跟踪初始化状态，避免在 `useEffect` 依赖中造成循环
- 只有在初始化完成后才处理 `onAuthStateChange` 事件
- 确保 `loading` 状态在正确的时机被设置

### 2. ProtectedRoute 的路由守卫问题 ⚠️

**问题位置**: `src/routes/router.tsx`

**问题原因**:
- 在认证状态快速变化时，可能导致路由不断重定向
- 如果 `loading` 状态在导航过程中变化，可能导致卡死
- 没有对状态变化进行优化，导致不必要的重渲染

**修复方案**:
- 使用明确的判断条件，避免在状态变化时产生歧义
- 优化渲染逻辑，减少不必要的重渲染

### 3. Login 页面的双重导航问题 ⚠️

**问题位置**: `src/pages/Login.tsx`

**问题原因**:
- `useEffect` 监听 `user` 变化并导航
- `signIn` 成功后也直接导航
- 这可能导致双重导航或导航冲突，特别是在登录后立即切换导航时

**修复方案**:
- 移除 `signIn` 中的直接导航，统一由 `useEffect` 处理
- 在 `useEffect` 中添加防抖机制，使用 `setTimeout` 确保导航在下一个事件循环中执行
- 添加 `loading` 检查，确保只在认证完成后导航

### 4. MainMenu 的导航优化 ✅

**问题位置**: `src/ui/MainMenu.tsx`

**问题原因**:
- 点击当前路径时也会触发导航，导致不必要的重渲染

**修复方案**:
- 添加路径检查，如果点击的是当前路径，不进行导航

## 修复内容总结

- ✅ 优化 `signIn` 方法，确保状态更新的一致性

### 2. ProtectedRoute 修复 (`src/routes/router.tsx`)
- ✅ 使用明确的判断条件，避免状态歧义

- ✅ 移除 `signIn` 中的直接导航

## 技术细节

### 竞态条件处理
使用 `useRef` 来跟踪初始化状态，避免了在 `useEffect` 依赖数组中包含状态导致的循环问题。

### 导航防抖
使用 `setTimeout` 将导航操作延迟到下一个事件循环，确保状态更新完成后再进行导航。

### 状态同步
确保 `onAuthStateChange` 和 `getUser()` 的状态更新不会冲突，通过初始化标志来控制处理时机。
