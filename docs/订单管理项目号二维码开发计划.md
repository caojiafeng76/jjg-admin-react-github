# 订单管理项目号二维码开发计划

## 一、需求背景

当前订单管理页面已经展示交货日期、客户、项目号、产品型号等核心订单字段，但现场或跨端查看订单时，仍需要先进入系统、再搜索项目号，操作链路较长。

本次计划是在订单管理表格中，为每条订单增加一个二维码入口，方便用户直接扫码获取该订单的关键信息。

## 二、目标

### 2.1 业务目标

- 在订单管理列表中，为每条订单生成唯一二维码。
- 用户扫描二维码后，可以直接查看该订单信息。
- 二维码列放在表格中，位置在交货日期之前，同时也位于项目号之前。
- 方案尽量复用现有订单数据，不引入重复维护的数据源。

### 2.2 交互目标

- 列表中一眼可见二维码入口，便于打印、截图或现场扫码。
- 扫码后展示的是适合手机查看的订单只读信息页，而不是后台复杂编辑页。
- 用户无需手动输入项目号，即可完成订单识别。

## 三、现状分析

结合当前项目结构，订单管理对应的是车间订单模块，订单列表数据来自 `sales_orders`。

当前列表已包含以下关键信息：

- 交货日期
- 客户
- 项目号
- 状态
- 产品型号
- 客户型号
- 长度
- 支数

当前仓库中暂无二维码相关实现，也没有现成的扫码订单详情页，因此本需求不只是“在表格里显示二维码”，还需要补齐扫码落地页与访问控制方案。

## 四、需求范围

### 4.1 本次纳入范围

- 订单管理列表新增二维码列。
- 为每条订单生成可扫码识别的唯一二维码。
- 提供扫码后的订单信息展示页或轻量详情页。
- 明确登录态、权限校验与访问边界。
- 完成移动端可读的订单信息展示。

### 4.2 本次不纳入范围

- 不做订单编辑能力下放到扫码页。
- 不做客户外部门户。
- 不做批量打印二维码标签。
- 不做离线扫码识别。

## 五、推荐方案

### 5.1 二维码内容

建议二维码承载的不是纯项目号文本，而是“订单扫码访问链接”。

推荐格式：

- 二维码内容为订单详情访问地址
- 地址中携带订单唯一标识，例如订单 ID
- 不建议直接把完整订单信息写入二维码

推荐原因：

- 避免二维码内容过长，影响扫码稳定性。
- 避免在二维码中暴露过多业务数据。
- 后续如需调整订单展示字段，只需要调整扫码详情页，不需要重建历史二维码格式。

### 5.2 扫码打开方式

建议采用“扫码进入订单只读页”的方式。

推荐链路：

1. 用户扫描二维码。
2. 系统打开订单详情只读页。
3. 若用户已登录且有权限，直接展示订单信息。
4. 若用户未登录，则先跳转登录，登录成功后回跳该订单详情页。

### 5.3 详情页展示内容

扫码页建议首期展示以下信息：

- 项目号
- 状态
- 交货日期
- 客户
- 产品型号
- 客户型号
- 长度
- 支数
- 表面处理
- 颜色
- 包装方式
- 材质
- 料号
- 最近更新时间

如果后续车间现场有进一步需求，可在二期增加：

- 生产进度摘要
- 已出库数量
- 相关生产工单链接

## 六、页面与交互调整

### 6.1 表格列调整

当前关键列顺序中，交货日期位于前侧，项目号位于其后。

本次调整后，建议顺序为：

- 序号
- 二维码
- 交货日期
- 客户
- 项目号
- 状态
- 其他原有字段保持不变

说明：

- 这满足“二维码在项目号前面、在交货日期之前”的要求。
- 二维码列建议固定在左侧，保证横向滚动后仍可见。

### 6.2 二维码显示方式

建议列表中显示小尺寸二维码缩略图，并配合以下体验设计：

- 默认显示可扫码的二维码图形
- 鼠标悬停时可显示更清晰的预览
- 点击二维码可打开大图或直接打开订单详情
- 列宽控制在不明显挤压现有核心字段的范围内

### 6.3 手机端扫码页要求

- 首屏直接展示订单核心信息
- 字体与间距适合手机浏览
- 不依赖复杂表格横向滚动
- 明确显示订单状态与项目号
- 若无权限或订单不存在，要给出明确提示

## 七、数据与接口规划

### 7.1 数据来源

优先复用现有 `sales_orders` 数据，不新增冗余订单表。

### 7.2 是否需要新增字段

首选方案：不新增二维码图片字段。

原因：

- 二维码可按订单 ID 或访问 URL 动态生成。
- 避免订单数据更新后，二维码图片仍需额外维护。
- 降低数据库存储成本与同步复杂度。

### 7.3 需要新增的能力

建议新增以下能力：

- 订单扫码详情查询能力
- 只读订单详情返回结构
- 登录后回跳扫码页能力

如果后续需要支持外部无登录访问，则再补充：

- 订单二维码访问令牌
- 令牌有效期
- 令牌失效与重发机制

## 八、权限与安全设计

### 8.1 首期推荐权限策略

首期建议按“内部用户扫码查看”设计，即：

- 二维码可以被任何人扫描
- 但真正查看订单详情时仍需登录
- 登录后继续沿用现有角色与数据访问权限

### 8.2 不建议的首期方案

不建议首期直接开放“扫码即匿名查看全部订单详情”，原因如下：

- 订单信息涉及客户、型号、料号等业务敏感数据
- 二维码一旦外流，难以控制传播范围
- 后续回收权限成本较高

### 8.3 异常场景

扫码页需要覆盖以下异常：

- 订单不存在
- 订单已删除
- 当前账号无权限查看
- 链接参数无效
- 登录过期

## 九、实施拆分

### 阶段一：方案确认

- 确认二维码是否仅面向内部员工使用
- 确认扫码后展示字段清单
- 确认是否需要独立移动端只读页
- 确认二维码列展示尺寸与表格列宽

### 阶段二：路由与访问链路

- 设计扫码访问地址规则
- 设计登录拦截与登录后回跳逻辑
- 规划订单只读详情页入口

### 阶段三：列表展示

- 在订单管理表格左侧新增二维码列
- 将二维码列插入到交货日期之前
- 保证固定列与横向滚动行为正常
- 验证列表分页、筛选、选中状态不受影响

### 阶段四：扫码详情页

- 输出适合手机查看的订单详情布局
- 接入订单详情查询
- 处理无权限、无数据、加载失败等状态

### 阶段五：联调与验收

- 后台列表二维码显示正确
- 扫码后可以进入目标订单
- 登录与权限逻辑正确
- 手机端阅读体验可接受

## 十、验收标准

### 10.1 列表验收

- 订单管理表格新增二维码列
- 二维码列位于交货日期之前
- 二维码列也位于项目号之前
- 列表分页、筛选、排序、选中功能正常

### 10.2 扫码验收

- 任意有效订单都能生成唯一二维码
- 使用微信、企业微信或系统相机扫码后可识别
- 已登录用户扫码后可直接看到订单详情
- 未登录用户扫码后会先登录，再回到目标订单详情

### 10.3 安全验收

- 无权限用户不能查看不应访问的订单
- 无效链接不会暴露系统异常信息
- 订单删除后，原二维码链接返回友好提示

## 十一、风险与注意事项

### 11.1 主要风险

- 如果直接把后台详情页暴露给手机端，阅读体验可能较差。
- 如果二维码编码的是纯项目号而不是唯一地址，后续详情页能力扩展会受限。
- 如果存在外网扫码场景，登录态与访问域名需要提前规划。
- 表格左侧固定列较多时，新增二维码列可能压缩现有可视区域。

### 11.2 对策建议

- 优先做独立只读扫码页，而不是复用后台编辑页。
- 优先编码访问链接，不直接编码业务详情。
- 首期按内部登录访问控制，避免一次性把匿名外部访问做复杂。
- 先确定二维码尺寸，再微调交货日期、客户、项目号等列宽。

## 十二、建议的交付物

本需求完成后，建议交付以下成果：

- 订单管理二维码功能说明
- 扫码只读页原型或页面说明
- 权限说明与异常场景说明
- 验收清单

## 十三、结论

本需求表面上是“在项目号前加一个二维码”，实质上包含三部分能力：

- 列表中稳定展示二维码
- 扫码后准确定位订单
- 在移动端安全地展示订单信息

建议首期采用“订单管理列表新增二维码列 + 扫码进入登录受控的订单只读详情页”的方案。这个方案改动边界清晰、风险可控，也最符合当前后台项目的权限体系，适合作为第一版实施路径。
