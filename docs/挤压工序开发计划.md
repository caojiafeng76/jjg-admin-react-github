# 挤压工序开发计划

> **实施说明：** 本文档用于指导挤压工序功能开发。实施时应按任务顺序推进，每完成一个阶段即执行对应验证，不应跳过数据库权限、缓存联动和订单现状回归。

**目标：** 新增独立的挤压工序生产记录功能，支持按日期、挤压机和班组批量录入项目明细，并将换算后的理论支数接入订单现状。

**架构：** 采用“挤压生产单主表 + 挤压生产明细表”的主从结构。主表记录日期、设备、班组长和班别，明细表记录项目号快照、铝棒投入、实际产出、损耗和派生指标；订单现状通过 Supabase RPC 按项目号聚合理论支数。

**技术栈：** React 19、TypeScript、Ant Design 6、TanStack Query 5、Supabase/PostgreSQL、Vitest。

## 1. 背景与范围

### 1.1 业务背景

当前系统已有生产工单、精切转移单和订单现状功能，但缺少挤压工序的独立生产台账。挤压记录包含设备、班组、铝棒投入、实际产出、损耗和成材率等专属字段，不适合直接复用通用生产工单明细。

本功能需要：

- 参考生产工单的“表头 + 多条明细”录入方式。
- 参考精切转移单的列表、详情、审核和缓存联动方式。
- 根据项目号自动带出订单型号、要求长度、理论米重和材质。
- 统一计算理论支数、理论支重、产出重量和成材率。
- 将理论支数汇总到订单现状的“挤压”列。

### 1.2 第一阶段范围

- 桌面端挤压生产记录列表。
- 新增、编辑、详情和删除。
- 一张生产单录入多条项目明细。
- 项目号选择和订单字段自动带出。
- 设备、班组长和班别录入。
- 派生字段实时预览与数据库统一计算。
- 页面权限、操作权限和 RLS。
- 订单现状聚合及明细弹窗。

### 1.3 暂不纳入第一阶段

- 员工手机端扫码录入。
- Excel 历史数据批量导入。
- 挤压排产。
- 铝棒库存扣减。
- 与设备运行时间自动关联。
- 自定义成材率预警规则。

## 2. 方案选择

### 2.1 推荐方案：主表 + 明细表

主表保存一张班次生产单的公共字段，明细表保存每个项目号的投入产出数据。

优点：

- 与生产工单现有交互模式一致。
- 日期、设备、班组长和班别无需在每行重复录入。
- 适合一个班次连续录入多个项目号。
- 后续增加审核、导出和移动端时扩展空间更好。

### 2.2 未采用方案

#### 单表台账

实现简单，但公共字段大量重复，整班编辑和审核不方便。

#### 复用 `production_order_items`

不采用。挤压的重量、铝棒、损耗和成材率规则与通用生产工单差异明显，复用会增加大量空字段和特殊判断，并容易影响现有订单现状岗位统计。

## 4. 计算规则

所有长度参与计算前必须统一单位。订单要求长度使用毫米，实际产出长度按图片使用米，因此计算理论支数时必须先将实际产出长度转换为毫米。

### 4.1 理论支数

```text
理论支数 =
floor(
  实际产出长度(m) × 1000
  ÷ 订单要求长度(mm)
  × 实际产出数量
)
```

必须对最终结果向下取整，不能先对单支可切数量取整。

示例：

```text
实际产出长度 = 6.05m
订单要求长度 = 200mm
实际产出数量 = 5支

理论支数 = floor(6.05 × 1000 ÷ 200 × 5)
          = floor(151.25)
          = 151支
```

边界规则：

- `order_required_length_mm <= 0`：禁止保存，提示“订单要求长度必须大于 0”。
- `actual_output_length_m <= 0`：禁止保存。
- `actual_output_quantity < 0`：禁止保存。
- `actual_output_quantity = 0`：理论支数为 0。
- 理论支数不得手工编辑。

### 4.2 理论支重

```text
理论支重(kg/支) = 理论米重(kg/m) × 实际产出长度(m)
```

### 4.3 实际产出重量

```text
实际产出重量(kg) = 实际支重(kg/支) × 实际产出数量
```

### 4.4 成材率

```text
成材率(%) = 实际产出重量 ÷ 铝棒投入重量 × 100
```

当铝棒投入重量小于等于 0 时禁止保存，避免除零。

### 4.5 铝棒投入重量

图片数据大致符合圆柱体积、铝密度和数量的换算，但正式实施前需要确认：

- 固定采用哪一个铝密度值。
- 投入重量是系统理论计算值还是现场称重值。
- 理论值是否允许人工修正。
- 小数保留位数和舍入方式。

第一阶段建议将 `aluminum_input_weight_kg` 作为最终保存值；若业务确认由系统计算，再增加统一计算函数，并允许明确记录“理论值/称重值”的来源。

### 4.6 计算职责

- 数据库：最终计算口径和约束的来源。
- 服务层：提交前标准化数值并返回明确中文错误。
- 表单：实时预览，不作为最终可信来源。
- 列表、详情、导出和订单现状：只读取统一结果，不重复编写公式。

## 9. 订单现状接入

### 9.1 聚合口径

订单现状中的“挤压”数量推荐使用：

```text
同项目号所有有效挤压明细的 theoretical_quantity 合计
```

原因：

- `actual_output_quantity` 表示挤压长料支数。
- 订单数量通常按订单要求长度对应的成品件数统计。

## 11. 核心验收用例

### 11.1 理论支数

| 实际产出长度 | 订单要求长度 | 实际产出数量 | 预期理论支数 |
| ------------ | ------------ | ------------ | ------------ |
| `6.05m`      | `200mm`      | `5`          | `151`        |
| `5.275m`     | `200mm`      | `158`        | `4167`       |
| `5.275m`     | `200mm`      | `0`          | `0`          |

第二行计算：

```text
floor(5.275 × 1000 ÷ 200 × 158)
= floor(4167.25)
= 4167
```

### 11.2 重量和成材率

使用图片首行数据：

```text
实际支重 = 5.23kg/支
实际产出数量 = 158支
实际产出重量 = 5.23 × 158 = 826.34kg

铝棒投入重量 = 970.56kg
成材率 = 826.34 ÷ 970.56 × 100% = 85.14%
```

### 11.3 订单现状

- 新建挤压明细后，对应项目号“挤压”列增加理论支数。
- 编辑长度、订单要求长度或实际产出数量后，聚合值重新计算。
- 删除挤压明细后，聚合值减少。
- 点击“挤压”列可查看所有来源明细。
- 挤压列不得重复叠加通用生产工单中名为“挤压”的岗位产量。

### 11.4 权限

- 管理员可以执行全部操作。
- 仅有页面权限的用户只能查看。
- 拥有创建权限但无编辑权限的用户不能修改历史记录。
- 无审核权限的用户不能审核或反审核。
- 无页面权限的用户直接调用 Supabase API 也无法读取数据。

## 12. 风险与控制

### 单位混用

实际产出长度为米，订单要求长度为毫米。必须在统一计算函数中完成换算，并通过测试锁定。

### 历史订单变更

理论支数依赖订单要求长度，因此必须保存订单长度快照，历史记录不能随订单修改自动漂移。

### 重复统计

订单现状已有通用生产工单岗位列。如果岗位基础资料中也存在“挤压”，接入独立挤压台账后必须排除通用“挤压”岗位列，避免出现两个挤压列或重复产量。

### 非原子写入

主表和多条明细必须在事务或 RPC 中一次写入。任一明细失败时整张生产单回滚。

### 缓存不刷新

所有挤压 Mutation 必须失效 `extrusion-productions` 和 `order-status-dashboard` 两组查询键。
