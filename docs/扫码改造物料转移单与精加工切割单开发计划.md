# 扫码改造物料转移单与精加工切割单开发计划

## 1. 背景

当前系统已经完成以下基础能力：

- 物料转移单与精加工切割单都已具备独立路由、桌面端 CRUD、员工端移动列表
- 两个表单都已接入 ProjectNoScanButton，可扫描项目号并自动带出客户、型号、长度、客户型号
- 员工端移动容器已经存在统一底部导航，并已新增扫码入口与扫码导航页
- 生产工单已经完成一版“扫码入口 -> 独立移动端页面 -> 底部弹层选择 -> 固定底部提交”的改造

目前这两个模块在员工端仍以“列表页点击新建 -> 打开移动端 Modal 表单”为主，虽然能录入，但仍有几个明显问题：

- 扫码入口不够集中，员工需要先进入业务列表页再点击新建
- 长表单放在 Modal 内，手机键盘、滚动、选择器弹层容易互相干扰
- 项目号虽然支持扫码，但整体交互仍偏桌面模式，不是“扫码优先”
- 两个模块都在重复做移动端弹窗适配，后续维护成本会继续上升

本次目标不是重写业务逻辑，而是仿照生产工单扫码方案，把物料转移单与精加工切割单改造成真正适合员工手机端的扫码录入流程。

## 2. 改造目标

### 2.1 业务目标

- 员工可从统一扫码导航页直接进入“扫码新增物料转移单”与“扫码新增精加工切割单”
- 扫码后自动填充项目号，并带出关联物料信息
- 员工仍可在未扫码时手动选择项目号继续录入
- 提交时继续沿用现有单据模型：每次提交创建一张新的业务单据，不引入生产工单那种“按天归并主表”的逻辑

### 2.2 交互目标

- 员工端扫码录入改为独立页面，不再依赖 Modal 承载完整长表单
- 项目号、接收车间、接收人、操作人等选择项在移动端统一改为底部弹层
- 页面固定底部提交按钮，避免长表单滚动到底部才能提交
- 保留管理员与桌面端当前交互，避免影响既有 CRUD 流程

## 3. 现状分析

## 3.1 物料转移单

当前 material-transfer 模块已经具备：

- 员工端与管理员共用同一路由
- 员工端列表样式已单独适配 MaterialTransferMobileList
- 员工端点击“新建”时仍打开 MaterialTransferForm Modal
- MaterialTransferForm 已接入 ProjectNoScanButton
- 表单字段已经完整覆盖：
  - 项目号
  - 客户
  - 型号
  - 长度(mm)
  - 客户型号
  - 转移数量
  - 操作人
  - 接收车间
  - 接收人
  - 当班负责人
  - 检验人
  - 备注

结论：物料转移单不缺扫码能力，缺的是“扫码优先入口 + 移动端独立页面 + 底部弹层交互”。

## 3.2 精加工切割单

当前 precision-finishing-cutting 模块已经具备：

- 员工端与管理员共用同一路由
- 员工端列表样式已单独适配 PrecisionFinishingCuttingMobileList
- 员工端点击“新建”时仍打开 PrecisionFinishingCuttingForm Modal
- PrecisionFinishingCuttingForm 已接入 ProjectNoScanButton
- 表单字段已经完整覆盖：
  - 项目号
  - 客户
  - 型号
  - 长度(mm)
  - 客户型号
  - 长料长度(mm)
  - 长料数量
  - 原料不良数
  - 加工不良数
  - 外协不良数
  - 不良原因
  - 外协不良原因
  - 外协单位
  - 转移数量
  - 责任工序
  - 工序负责人
  - 操作人
  - 接收车间
  - 接收人
  - 检验人
  - 备注

结论：精加工切割单比物料转移单字段更多，更不适合继续放在移动端 Modal 中完成录入，因此优先级甚至更高。

## 4. 方案原则

本次改造建议遵循以下原则：

- 不改现有表结构，不新增新的单据模型
- 不改管理员桌面端主流程
- 只新增员工端扫码录入路径
- 优先抽离共享的移动端扫描表单壳子与底部弹层能力
- 项目号扫码解析继续复用现有 ProjectNoScanButton

## 5. 路由与入口方案

## 5.1 入口位置

建议继续复用现有扫码导航页，而不是把物料转移单、精加工切割单直接塞到底部导航。

当前已确认按这个方案推进。

推荐原因：

- 底部导航位数已经有限，继续加业务入口会过于拥挤
- 当前扫码导航页已经被设计为“后续可继续增加多个按钮”的入口页
- 生产工单、物料转移单、精加工切割单都属于“扫码动作触发的录入页”，适合统一收口

建议在现有扫码导航页新增两个按钮：

- 扫码物料转移单
- 扫码精加工切割单

当前已确认先把这两个入口接到扫码导航页，再继续补各自扫描页。

## 5.2 路由建议

建议新增两个独立移动端路由：

- /material-transfer/scan
- /precision-finishing-cutting/scan

页面命名建议：

- MobileMaterialTransferScanPage.tsx
- MobilePrecisionFinishingCuttingScanPage.tsx

## 5.3 现有页面行为调整

建议员工端在列表页点击“新建”时，也跳转到对应 scan 页面，而不是继续打开移动端 Modal。

这样可以保证入口统一：

- 从扫码导航页进入，可以直接扫码
- 从业务列表页进入，也落到同一个移动端录入页面

管理员与桌面端保留原有 Modal 表单，不做破坏性调整。

## 6. 页面结构方案

两个扫描页可以使用一致的页面骨架：

- 顶部标题与返回按钮
- 扫码主按钮
- 最近识别结果提示区
- 物料信息展示卡片
- 业务字段表单区
- 底部固定提交按钮

推荐抽出共享布局层，例如：

- MobileScanFormLayout
- MobileBottomSelectSheet
- MobileProjectSummaryCard

这样两个页面主要只保留各自字段差异，避免后续继续复制样式与交互逻辑。

## 7. 扫码与自动带值方案

## 7.1 可直接复用

两个模块都已经在表单内实现：

- 扫码识别项目号
- 把扫描结果放入本地 scannedProjectDataMap
- 将扫码得到的项目号补进项目号下拉候选
- 自动带出客户、型号、长度、客户型号

因此扫描页不需要重新设计扫码算法，只需要把这套逻辑从 Modal 使用场景转移到页面使用场景。

## 7.2 扫码页期望行为

进入扫描页后，用户可选择两条路径：

- 直接点击扫码按钮，完成自动回填
- 不扫码，手动打开底部弹层选择项目号

当项目号变化时，应继续沿用现有逻辑：

- 更新客户
- 更新型号
- 更新长度(mm)
- 更新客户型号

## 8. 移动端选择交互方案

当前两个表单在移动端仍主要依赖 Select、AutoComplete 的弹层，虽然 width 已做移动适配，但交互本质仍是桌面下拉。

建议改为底部弹层交互的字段：

### 8.1 两个页面共用

- 项目号
- 操作人
- 接收车间
- 接收人
- 检验人候选值若后续需要固定化，也可纳入同一方案

### 8.2 物料转移单可选扩展

- 当班负责人

### 8.3 精加工切割单可选扩展

- 责任工序
- 工序负责人

若责任工序与工序负责人当前仍然是自由输入，则第一版可以保持 Input，优先只把高频选择型字段改成底部弹层。

## 9. 提交流程方案

## 9.1 物料转移单

提交时继续复用现有 createMaterialTransfer 流程，重点保持以下规则：

- employee 角色默认绑定当前员工为操作人
- uploaded_by_name 继续记录当前登录人
- is_audited 在员工端强制为 false
- 校验转移数量必须大于 0

本次不需要像生产工单那样按日期复用主单，因为 material_transfers 本身就是单条独立记录。

## 9.2 精加工切割单

提交时继续复用现有 createPrecisionFinishingCutting 流程，重点保持以下规则：

- employee 角色默认绑定当前员工为操作人
- uploaded_by_name 继续记录当前登录人
- is_audited 在员工端强制为 false
- 原料不良数、加工不良数、外协不良数默认值保持兼容

同样不引入主从归并逻辑，每次提交就是一张新的精加工切割单。

## 12. 开发阶段建议

### 第一阶段：扫码导航页与路由先行

- 先在现有扫码导航页补两个新入口
- 先注册 /material-transfer/scan 与 /precision-finishing-cutting/scan
- 先完成 lazyPages、router、标题映射等骨架接入
- 先把“统一入口 + 独立路由”的框架搭起来，再进入表单页细化

这样可以先把整体访问路径跑通，避免两个业务页面各自开发后再回头重接导航。

### 第二阶段：方案补充确认

- 确认两个新入口都挂在现有扫码导航页
- 确认员工端列表页“新建”是否统一跳转 scan 页面
- 确认责任工序、工序负责人是否保持手工输入

### 第三阶段：物料转移单扫描页

- 新建 MobileMaterialTransferScanPage
- 先把物料转移单从移动端 Modal 迁到独立页面
- 先打通扫码、手动选项目号、固定底部提交按钮
- 优先验证物料转移单的底部弹层交互

### 第四阶段：精加工切割单扫描页

- 新建 MobilePrecisionFinishingCuttingScanPage
- 再把精加工切割单从移动端 Modal 迁到独立页面
- 复用前一阶段沉淀的页面壳子和底部弹层能力
- 重点处理长表单分组与固定底部提交

### 第五阶段：共享交互收口

- 项目号改成底部弹层
- 选择类字段改成底部弹层
- 对两个页面统一底部弹层体验
- 对两个页面统一固定底部提交栏

### 第六阶段：联调回归

- 回归管理员桌面端新增/编辑
- 回归员工端列表查询与编辑
- 验证扫码与手动录入都可正常创建单据

## 13. 风险与注意事项

- 两个模块当前员工端“新建”直接依赖 Modal，改成页面后要避免误伤桌面端行为
- 精加工切割单字段较多，若一次性全量改为底部弹层，容易拉长开发周期，应优先处理高频选择型字段
- AutoComplete 改造成底部弹层后，要注意保留“可搜索 + 可直接选择 + 可手输”的组合体验
- 若继续依赖 antd 默认下拉浮层，在移动端键盘场景下仍可能出现遮挡和焦点问题，因此建议优先落地底部弹层

## 14. 预期结果

改造完成后，员工端将形成统一的扫码录入闭环：

- 从扫码导航页进入目标业务
- 扫码自动带出项目信息
- 在移动端独立页面中完成录入
- 使用底部弹层完成高频选择操作
- 固定底部提交，减少滚动与误触

这样可以让物料转移单、精加工切割单与生产工单在员工手机端保持一致的扫描体验与交互心智。
