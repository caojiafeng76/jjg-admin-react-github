## 二、数据库设计决策

### 2.1 是否新增运行时间数据表？

**结论：不新增**。

运行时间是派生值，可通过以下关系实时计算：

$$\text{运行时间（秒）} = \texttt{incoming\_qualified\_quantity} \times \texttt{theoretical\_seconds}$$

- `incoming_qualified_quantity` 来自 `production_order_items`
- `theoretical_seconds`（工序理论加工时间）来自 `process_standards`

存储策略：在 `production_order_items` 中存储 `theoretical_seconds` **快照**（与现有 `standard_seconds` 快照字段同理），避免工艺标准更新导致历史运行时间失真。

### 2.2 新增字段（一个 Migration 文件）

目标表：`production_order_items`

| 新字段 | 类型 | 说明 |
|--------|------|------|
| `machine_equipment_id` | `uuid NULL` | FK → `machine_equipment_maintenances.id`，`ON DELETE SET NULL`；`NULL` = "无" |
| `theoretical_seconds` | `float8 NOT NULL DEFAULT 0` | 理论加工时间快照（秒） |

历史数据处理：
- `machine_equipment_id` 默认 `NULL`（= "无"），无需迁移逻辑
- `theoretical_seconds` 在 Migration 中追加一条 `UPDATE ... FROM process_standards` 尽力回填，找不到匹配的保持 `0`

### 2.3 是否创建数据库 VIEW？

**建议创建** `v_machine_runtime_items` VIEW，简化前端服务层的多表 JOIN 查询（`production_order_items` JOIN `production_orders` JOIN `employees` JOIN `machine_equipment_maintenances`）。

> 注意：Supabase VIEW 的 TypeScript 类型需在 `database.types.ts` 中手动扩展（因为该文件由 CLI 生成，VIEW 类型不一定自动完整）。

## 三、开发任务清单（按依赖顺序）

### Phase 1 — 数据库层

#### T1. 创建 Supabase Migration 文件

**文件**：`supabase/migrations/20260408_add_machine_runtime_fields.sql`

内容：

1. 添加 `machine_equipment_id` 列（FK，可为空）
2. 添加 `theoretical_seconds` 列（快照，默认 0）
3. 历史数据回填：从 `process_standards` JOIN 补全 `theoretical_seconds`（best-effort）
4. 创建 VIEW `v_machine_runtime_items`：

```sql
CREATE OR REPLACE VIEW v_machine_runtime_items AS
SELECT
  poi.id,
  poi.order_id,
  poi.project_no,
  poi.product_model,
  poi.customer_model,
  poi.length_mm,
  poi.operation,
  poi.incoming_qualified_quantity,
  poi.theoretical_seconds,
  poi.machine_equipment_id,
  poi.incoming_qualified_quantity * poi.theoretical_seconds AS runtime_seconds,
  po.order_date,
  po.employee_id,
  e.name AS operator_name,
  me.unified_device_no,
  me.operation AS device_operation,
  me.machine_name
FROM production_order_items poi
LEFT JOIN production_orders po ON poi.order_id = po.id
LEFT JOIN employees e ON po.employee_id = e.id
LEFT JOIN machine_equipment_maintenances me ON poi.machine_equipment_id = me.id;
```

#### T2. 更新 `database.types.ts`

执行 `npx supabase gen types typescript` 重新生成，或在 `production_order_items` 的 Row / Insert / Update 类型中手动追加两个新字段（仅临时过渡用）。

> **注意**：项目约定该文件由 CLI 生成，禁止持久手动修改。Migration 推送后应及时执行 CLI 同步类型。

### Phase 2 — 服务层

#### T3. 更新 `apiProductionOrderItems.ts`

`ProductionOrderItem` 类型通过 `Database['public']['Tables']['production_order_items']['Row']` 派生，T2 完成后自动包含新字段，**无需手动改服务层类型**。

若 Insert / Update 函数签名有手动 Pick，需加入新字段。

#### T4. 更新 `apiMachineEquipmentMaintenances.ts`

新增 `getMachineEquipmentOptions()` 函数：

- 查询全量设备列表（用于工序明细表单下拉）
- 返回字段：`{ id, unified_device_no, operation, machine_name }`
- 按 `unified_device_no` 排序

#### T5. 新建 `apiMachineRuntime.ts`

新增两个查询函数：

| 函数 | 说明 |
|------|------|
| `getMachineRuntimeItems(filters)` | 明细列表（带筛选、分页），基于 VIEW |
| `getMachineRuntimeSummary(filters)` | 汇总统计（GROUP BY 设备），返回每台设备的运行总时间 |

`MachineRuntimeFilters` 包含：`dateFrom`、`dateTo`、`unifiedDeviceNo`、`deviceOperation`、`machineName`

### Phase 3 — 功能 1 前端

#### T6. 新建 `src/features/production-order/useMachineEquipmentOptions.ts`

封装设备选项查询 Hook，`staleTime` 使用 `queryConfig.detail`（10 分钟），设备列表变化频率低。

#### T7. 修改 `ProductionOrderItemForm.tsx`

新增 `machine_equipment_id` 字段：

- **控件**：`<Select showSearch />`
- **选项列表**：
  - 第 0 项：`value = null`，显示文本"无"
  - 后续项：来自 `useMachineEquipmentOptions()`
- **选项渲染**（`optionRender`）双行格式：
  ```
  第一行（正常字号）：统一设备编号
  第二行（小字 / 灰色）：工序名   机器名称
  ```
- **选中后显示**（`optionLabelProp`）：统一设备编号（选"无"则显示"无"）
- **搜索范围**：`unified_device_no`、`operation`、`machine_name` 均可匹配
- **联动**：选定工序后预筛选设备列表，仅显示 `device_operation` 匹配的设备
- **校验规则**：
  - `value === undefined`（未选择）→ 报错"请选择机器编号"
  - `value === null`（选了"无"）→ 合法，可提交
- **theoretical_seconds 带入**：选定工序后自动从 `process_standards` 带入 `theoretical_seconds`（仿照现有 `standard_seconds` 带入逻辑）

#### T8. 修改 `ProductionOrderItemTable.tsx`

- 在"工序"列右侧新增 **"机器编号"** 列：
  - 查询时用 Supabase 嵌套 select 带回设备信息：`select('*, machine_equipment_maintenances!machine_equipment_id(unified_device_no, operation, machine_name)')`
  - `machine_equipment_id === null` 时显示"无"
  - 在 `apiProductionOrderItems.ts` 定义扩展类型 `ProductionOrderItemWithMachine`

- 在末尾新增 **"运行时间"** 列：
  - 计算：`incoming_qualified_quantity × theoretical_seconds ÷ 3600`
  - 显示格式：`X.XX h`（保留两位小数）
  - 值为 0 时显示 `—`

## 四、风险与注意事项

| 风险 | 说明 | 应对措施 |
|------|------|----------|
| "必填"与"选无"的矛盾 | `null` 是合法值，但"未选择"是非法状态 | Form validator 判断 `value === undefined` 时报错；`value === null` 视为"选了无"，正常提交 |
| `database.types.ts` 更新 | 项目约定 CLI 生成，不应手动持久修改 | Migration 推送后执行 `npx supabase gen types typescript --local > src/services/database.types.ts` 重新生成 |
| 历史运行时间为 0 | 旧记录 `theoretical_seconds = 0`，回填不完整时运行时间显示 0 | 表格中值为 0 时用 `—` 替代，不影响功能；新增记录会存入正确值 |
| 嵌套查询类型扩展 | Supabase JS v2 嵌套 select 的 TS 类型需手动扩展 | 在 `apiProductionOrderItems.ts` 定义 `ProductionOrderItemWithMachine` 扩展类型 |
| FK 级联策略 | 删除设备后关联工单条目的 `machine_equipment_id` 自动变 NULL | `ON DELETE SET NULL`，业务上可接受，显示回落为"无" |
| 运行时间页面查询性能 | 跨多表 JOIN + GROUP BY 在数据量大时可能慢 | 默认限制日期范围（如最近 30 天）；加 `machine_equipment_id IS NOT NULL` 过滤排除"无"设备记录 |
