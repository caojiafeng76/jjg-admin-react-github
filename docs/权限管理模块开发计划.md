## 一、目标与背景

### 现状

目前系统使用基于固定角色（`admin` / `employee` / `team_leader` / `precision_cutting_admin`）的访问控制：

- 菜单项在 `MainMenu.tsx` 中按角色类型硬编码
- 路由保护通过 `RoleProtectedRoute` 写死允许的角色列表
- 无法对同一角色的不同用户配置差异化权限
- 新增模块后必须手动修改多处文件才能接入权限控制

### 目标

| 需求       | 说明                                                           |
| ---------- | -------------------------------------------------------------- |
| 导航级权限 | 控制用户能看到哪些菜单项（PC 侧边栏 / 移动端底部 Tab）         |
| 页面级权限 | 控制用户能否访问某个路由页面（PC 页面 + 移动端 H5 页面均纳入） |
| 功能级权限 | 控制页面内按钮/操作的**显示/隐藏**或**启用/禁用**              |
| 字段级权限 | 控制表单/详情页中某个字段的**可见**、**只读**或**可编辑**状态  |
| 移动端权限 | 员工手机端（扫码、工单录入、日报等 H5 页面）独立纳入权限管控   |
| CRUD 管理  | 提供后台管理界面对角色和权限进行增删改查                       |
| 自动发现   | 新增模块时，其权限自动注册到系统，无需手动维护                 |

## 二、整体设计思路

### 2.1 权限模型：RBAC + 代码清单驱动

采用 **角色-权限（RBAC）** 模型，在现有角色体系上叠加细粒度权限层：

```
用户（Employee）
  └── 角色（AppRole）
        └── 权限集合（Permission Set）
              ├── 导航权限  （nav:   syney, mobile-workspace, ...）
              ├── 页面权限  （page:  syney-po-list, mobile-production-order, ...）
              ├── 功能权限  （feature: syney-po-list.create, production-order.close, ...）
              └── 字段权限  （field: production-order.work_hours.view/edit, ...）
```

权限体系同时覆盖两个访问面：

| 访问面        | 说明                                    | 典型用户                       |
| ------------- | --------------------------------------- | ------------------------------ |
| **PC 管理端** | 桌面浏览器，侧边栏导航，完整表格/表单   | admin、precision_cutting_admin |
| **移动端 H5** | 手机浏览器，底部 Tab 导航，触屏优化页面 | employee、team_leader          |

### 2.2 权限标识符规范

```
<scope>:<module>[.<field>].<action>
```

- `scope`：`nav` / `page` / `feature` / `field`
- `module`：模块路由标识（与路由 key 对齐），移动端模块以 `mobile-` 前缀区分
- `action`（`feature` 用）：`create` / `edit` / `delete` / `export` / `audit` / `close` 等
- `action`（`field` 用）：`view`（可见） / `edit`（可编辑，隐含可见）

示例：

| 权限标识符                                          | 含义                              |
| --------------------------------------------------- | --------------------------------- |
| `nav:syney`                                         | PC 导航：西尼菜单分组             |
| `nav:mobile-workspace`                              | 移动端导航：员工工作台 Tab        |
| `page:syney-po-list`                                | PC 页面：西尼订单列表             |
| `page:mobile-production-order`                      | 移动端页面：我的工单              |
| `page:mobile-scan-hub`                              | 移动端页面：扫码中心              |
| `feature:syney-po-list.create`                      | 功能：西尼订单 - 新建             |
| `feature:syney-po-list.delete`                      | 功能：西尼订单 - 删除             |
| `feature:employee-list.edit-role`                   | 功能：员工管理 - 编辑角色         |
| `field:production-order.work_hours.view`            | 字段：生产工单 - 工时（可见）     |
| `field:production-order.work_hours.edit`            | 字段：生产工单 - 工时（可编辑）   |
| `field:production-order.extra_qualified_hours.view` | 字段：生产工单 - 超产工时（可见） |

### 2.3 自动发现机制

每个功能模块目录下新增 `permissions.ts`，声明本模块所有权限。系统启动时由全局 **权限注册表**（`src/config/permissionRegistry.ts`）自动聚合所有模块权限，并与数据库同步（仅新增，不删除已存在的）。

```
src/features/syney/PoList/
├── permissions.ts   ← 新增：声明本模块权限清单
└── ...其他文件
```

当开发者新增模块后：

1. 在模块目录创建 `permissions.ts`
2. 在注册表文件中 `import` 该模块的权限
3. 系统自动将新权限写入数据库，无需手动操作管理后台

### 2.4 移动端权限设计原则

移动端 H5 页面（`src/features/*/Mobile*.tsx`）与 PC 页面属于不同访问面，遵循以下原则：

1. **独立 page 权限**：每个移动端页面有独立的 `page:mobile-*` 权限，与对应 PC 页面权限分开控制。例如管理员可以看 PC 生产工单，但不一定能进入移动端工单录入流程。
2. **共享 feature 权限**：移动端对同一业务数据的操作（如新建工单、提交日报）与 PC 端共用同一个 `feature:*` 权限，无需重复定义。
3. **移动端导航 nav**：移动端底部 Tab / 工作台快捷入口对应 `nav:mobile-*` 权限，独立于 PC 侧边栏的 `nav:*`。
4. **扫码页面**：扫码入口（ScanHub、扫码添加工单、物料转移扫码等）视为独立移动端功能，各有对应 `page:mobile-scan-*` 权限。

**现有移动端路由对应权限：**

| 路由路径                             | 组件                            | 对应权限 key                             |
| ------------------------------------ | ------------------------------- | ---------------------------------------- |
| `/scan`                              | ScanHub                         | `page:mobile-scan-hub`                   |
| `/production-order`（移动端视图）    | ProductionOrder（手机端）       | `page:mobile-production-order`           |
| `/production-order/create`           | MobileProductionOrderCreate     | `feature:mobile-production-order.create` |
| `/production-order/scan`             | ProductionOrderScan             | `page:mobile-scan-production-order`      |
| `/production-order/:id`              | MobileProductionOrderDetail     | `page:mobile-production-order`（复用）   |
| `/production-order/:id/edit`         | MobileProductionOrderEdit       | `feature:mobile-production-order.edit`   |
| `/production-daily-report`（移动端） | ProductionDailyReport（手机端） | `page:mobile-production-daily-report`    |
| `/material-transfer/scan`            | MaterialTransferScan            | `page:mobile-scan-material-transfer`     |
| `/precision-finishing-cutting/scan`  | PrecisionFinishingCuttingScan   | `page:mobile-scan-precision-finishing`   |

### 2.5 字段级权限设计

字段权限用于控制表单字段、详情列的**可见性**和**可编辑性**，适用于以下场景：

- 敏感字段（工资、成本、工时）只有特定角色/人员可见
- 某字段对普通用户只读，对管理员可编辑（如物料转移单的审核备注）
- 移动端表单隐藏 PC 端特有的管理字段（如成本系数）

#### 字段权限三态逻辑

字段的最终状态由**两个布尔权限**组合决定：

| `field:X.view` | `field:X.edit` | 最终状态   | 说明                                          |
| :------------: | :------------: | ---------- | --------------------------------------------- |
|       ✅       |       ✅       | **可编辑** | 显示且可输入                                  |
|       ✅       |       ❌       | **只读**   | 显示但不可修改（Input disabled / 纯文本展示） |
|       ❌       |       ❌       | **隐藏**   | 字段不渲染                                    |
|       ❌       |       ✅       | **可编辑** | edit 权限隐含 view，自动升级为可编辑          |

#### 字段权限 Hook 使用方式

```typescript
// 返回 'editable' | 'readonly' | 'hidden'
const state = useFieldPermission('production-order', 'work_hours')

// 在表单字段中使用
{state !== 'hidden' && (
  <Form.Item name="work_hours" label="工时">
    <InputNumber disabled={state === 'readonly'} />
  </Form.Item>
)}
```

#### 需纳入字段权限管控的优先字段清单

| 模块       | 字段                                     | 敏感原因     | 默认建议                        |
| ---------- | ---------------------------------------- | ------------ | ------------------------------- |
| 生产工单   | `work_hours`（工时）                     | 影响成本核算 | admin 可编辑，employee 可见只读 |
| 生产工单   | `extra_qualified_hours`（超产工时）      | 影响奖励计算 | admin 可编辑，team_leader 可见  |
| 成本核算   | `labor_cost`（人工成本）                 | 财务敏感     | 仅 admin 可见                   |
| 成本核算   | `labor_cost_coefficient`（人工费率系数） | 核算参数     | 仅 admin 可编辑                 |
| 员工管理   | `hourly_wage`（时薪）                    | 薪资敏感     | 仅 admin 可见                   |
| 员工管理   | `performance_score`（绩效）              | 薪资敏感     | admin 可编辑，本人只读          |
| 物料转移单 | `is_audited` / 审核字段                  | 审核权限     | 仅特定角色可操作                |
| 精切转移单 | `defect_reason`（不良原因）              | 责任认定     | admin/team_leader 可编辑        |

## 三、数据库设计

### 3.1 新增表

#### `permissions` 表（权限清单注册表）

```sql
CREATE TABLE permissions (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  key         text NOT NULL UNIQUE,           -- 权限标识符，如 feature:syney-po-list.create
  scope       text NOT NULL,                  -- nav | page | feature | field
  surface     text NOT NULL DEFAULT 'pc',     -- pc | mobile | both
  module      text NOT NULL,                  -- 模块标识，如 syney-po-list
  field_name  text,                           -- 字段名（仅 scope=field 使用），如 work_hours
  action      text,                           -- 操作名，如 create / delete / view / edit（page/nav 为 null）
  label       text NOT NULL,                  -- 显示名，如 "西尼订单 - 新建"
  description text,                           -- 补充说明
  created_at  timestamptz NOT NULL DEFAULT now()
);

-- 权限只能由代码注册，禁止前端直接 INSERT
ALTER TABLE permissions ENABLE ROW LEVEL SECURITY;
CREATE POLICY "admin can read permissions"
  ON permissions FOR SELECT TO authenticated
  USING (auth.jwt() ->> 'role' = 'admin' OR EXISTS (
    SELECT 1 FROM employees WHERE auth_uid = auth.uid() AND is_active = true
  ));
```

#### `role_permissions` 表（角色-权限分配）

```sql
CREATE TABLE role_permissions (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  role          text NOT NULL,                -- AppRole: admin | employee | team_leader | precision_cutting_admin
  permission_id uuid NOT NULL REFERENCES permissions(id) ON DELETE CASCADE,
  enabled       boolean NOT NULL DEFAULT true,
  created_at    timestamptz NOT NULL DEFAULT now(),
  updated_at    timestamptz NOT NULL DEFAULT now(),
  UNIQUE (role, permission_id)
);

-- 仅管理员可维护
ALTER TABLE role_permissions ENABLE ROW LEVEL SECURITY;
CREATE POLICY "admin can manage role_permissions"
  ON role_permissions FOR ALL TO authenticated
  USING (EXISTS (
    SELECT 1 FROM employees WHERE auth_uid = auth.uid() AND role = 'admin' AND is_active = true
  ));
CREATE POLICY "authenticated can read role_permissions"
  ON role_permissions FOR SELECT TO authenticated
  USING (true);
```

#### `user_permission_overrides` 表（用户级权限覆盖，可选）

> 用于对特定用户授予或收回超出其角色默认权限的访问能力。

```sql
CREATE TABLE user_permission_overrides (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  employee_id   uuid NOT NULL REFERENCES employees(id) ON DELETE CASCADE,
  permission_id uuid NOT NULL REFERENCES permissions(id) ON DELETE CASCADE,
  granted       boolean NOT NULL,            -- true=覆盖授权, false=覆盖撤销
  created_at    timestamptz NOT NULL DEFAULT now(),
  updated_at    timestamptz NOT NULL DEFAULT now(),
  UNIQUE (employee_id, permission_id)
);

ALTER TABLE user_permission_overrides ENABLE ROW LEVEL SECURITY;
CREATE POLICY "admin can manage user overrides"
  ON user_permission_overrides FOR ALL TO authenticated
  USING (EXISTS (
    SELECT 1 FROM employees WHERE auth_uid = auth.uid() AND role = 'admin' AND is_active = true
  ));
CREATE POLICY "user can read own overrides"
  ON user_permission_overrides FOR SELECT TO authenticated
  USING (employee_id IN (
    SELECT id FROM employees WHERE auth_uid = auth.uid()
  ));
```

### 3.2 数据库函数

#### 获取当前用户的完整权限集

```sql
CREATE OR REPLACE FUNCTION get_my_permissions()
RETURNS TABLE(key text, enabled boolean)
LANGUAGE plpgsql SECURITY DEFINER AS $$
DECLARE
  v_employee_id uuid;
  v_role text;
BEGIN
  SELECT id, role INTO v_employee_id, v_role
  FROM employees
  WHERE auth_uid = auth.uid() AND is_active = true;

  -- 角色权限 + 用户覆盖，覆盖优先
  RETURN QUERY
  SELECT
    p.key,
    COALESCE(upo.granted, rp.enabled) AS enabled
  FROM permissions p
  LEFT JOIN role_permissions rp ON rp.permission_id = p.id AND rp.role = v_role
  LEFT JOIN user_permission_overrides upo ON upo.permission_id = p.id AND upo.employee_id = v_employee_id
  WHERE rp.id IS NOT NULL OR upo.id IS NOT NULL;
END;
$$;
```

#### `<PermissionField>` 组件（字段级权限封装）

包裹 Form.Item，根据字段权限自动控制字段的渲染状态：

```tsx
// 自动处理 hidden / readonly / editable 三态
;<PermissionField module="production-order" field="work_hours">
  <Form.Item name="work_hours" label="工时">
    <InputNumber />
  </Form.Item>
</PermissionField>

// 等价于：
const state = useFieldPermission('production-order', 'work_hours')
// state === 'hidden'   → 不渲染
// state === 'readonly' → Form.Item 内的控件自动 disabled
// state === 'editable' → 正常渲染
```

> **注意**：`<PermissionField>` 内部通过 React cloneElement 给子控件注入 `disabled` 属性，  
> 对于无法接受 `disabled` 的自定义子组件，可直接使用 `useFieldPermission` hook 手动处理。

### 4.8 权限同步（自动发现）

在应用初始化时调用一次同步函数，将 `PERMISSION_REGISTRY` 中新增的权限写入数据库：

**`src/services/apiPermissions.ts`**

```typescript
// 仅管理员会触发，将代码中定义但数据库中不存在的权限批量 upsert
export async function syncPermissionRegistry(
  registry: PermissionDefinition[],

```

> 触发时机：管理员首次登录后，在 `PermissionContext` 中静默调用。非管理员不调用，避免权限问题。

### PC 端模块接入 Checklist

```
□ 1. 在模块目录创建 permissions.ts，声明所有 nav/page/feature/field 权限

□ 9. 管理员在权限管理后台为各角色配置新模块的默认权限
```

```
□ 12. 移动端表单中涉及敏感字段统一用 <PermissionField> 控制，
      module 使用 'mobile-<module>' 与 PC 端字段权限区分
□ 13. 若移动端需要独立的导航入口，在 MOBILE_NAV_ITEMS 中添加并关联 nav:mobile-* 权限
```

## 十、风险与约束

| 风险                                                    | 缓解措施                                                                           |
| ------------------------------------------------------- | ---------------------------------------------------------------------------------- |
| 现有角色分支（adminItems 等）与新权限系统并行期间不一致 | Phase 2 完成后废弃旧分支，过渡期兼容运行                                           |
| 权限加载增加首屏请求                                    | 使用 TanStack Query prefetch，随 `get_my_permissions()` 一次拉取                   |
| 数据库同步可能创建重复权限                              | `permissions.key` 字段设置 UNIQUE 约束，sync 使用 upsert                           |
| 管理员误删所有权限导致锁死                              | `admin` 角色硬编码保底权限（代码中兜底），不依赖数据库                             |
| 权限 key 重命名导致历史数据失效                         | 规范 key 命名，变更 key 时写 migration 更新历史数据                                |
| 移动端 `EMPLOYEE_SIDE_ROLES` 过渡期与新权限并存         | Phase 3c 改造后移除硬编码，过渡期两者均判断（OR 逻辑）                             |
| 字段权限未配置时的默认行为                              | `useFieldPermission` 默认返回 `'editable'`（保持现有体验），需显式配置才限制       |
| 字段 `disabled` 注入与 antd 受控组件冲突                | `<PermissionField>` 仅注入 `disabled` prop，不干预值绑定；复杂场景用 hook 手动处理 |
