# 成本核算开发计划

## 1. 目标

在现有“标准工时”能力上扩展为“成本核算”，满足以下诉求：

- 管理员可维护成本核算基础参数
- 系统按既定公式自动计算单件成本明细与合计
- 管理员可对成本核算数据执行完整 CRUD
- 管理员可导出选中数据为 Excel
- 不破坏现有生产工单对 `process_standards.standard_seconds` 的依赖

## 2. 现状评估

基于当前代码库，已有较好的改造基础：

- 路由仍使用 `standard-time-list`，页面实际实现位于 [src/features/workshop/StandardTimeList/index.tsx](src/features/workshop/StandardTimeList/index.tsx)
- 列表数据直接来自 `process_standards` 表，服务层位于 [src/services/apiStandardTimes.ts](src/services/apiStandardTimes.ts)
- 管理员已有批量勾选、创建、编辑、删除能力
- 班组长手机端当前仅维护 `theoretical_seconds`，该权限模型已形成既有约束
- 项目已具备通用导出按钮与多套 Excel 导出工具，可直接复用 [src/ui/ExportButton.tsx](src/ui/ExportButton.tsx)
- 菜单文案已经出现“成本核算”，但页头和模块内部文案仍大量保留“标准工时”，存在命名不一致

数据库和业务约束：

- `process_standards` 当前存在 `(model, operation)` 唯一约束
- `production_order_items` 通过 `(product_model, operation)` 外键依赖 `process_standards`
- `standard_seconds` 变化会通过数据库触发器同步到历史工单明细

结论：

- 最稳妥的 V1 方案，是继续以 `process_standards` 作为成本核算主表进行扩展
- 这样可以最大程度复用现有页面、查询、权限和工单关联逻辑
- 不建议新建独立页面和独立主表，否则要额外处理标准工时与成本核算之间的数据同步问题

## 3. 推荐数据设计

### 3.1 表设计原则

推荐直接扩展 `process_standards` 表，分为两类字段：

- 人工维护字段：管理员可编辑
- 自动计算字段：数据库统一计算，只读展示，不允许前端直接写入

### 3.2 人工维护字段

建议新增以下字段：

| 字段名 | 类型建议 | 默认值 | 单位 | 说明 |
| --- | --- | --- | --- | --- |
| labor_rate | numeric(10,4) | 0 | 元/小时 | 人工费率 |
| equipment_rate | numeric(10,4) | 0 | 元/小时 | 设备费率 |
| tool_rate | numeric(10,4) | 0 | 元/支 | 刀具费率 |
| cutting_fluid_rate | numeric(10,4) | 0 | 元/支 | 切削液费率 |
| fixture_rate | numeric(10,4) | 0 | 元/支 | 工装费率 |
| inspection_seconds | integer | 0 | 秒 | 检验工时 |
| daily_management_cost | numeric(12,2) | 0 | 元 | 日管理总费用 |
| daily_total_hours | numeric(10,2) | 0 | 小时 | 日总工时 |
| remark | text | null | - | 备注 |

说明：

- `inspection_seconds` 建议使用整数，和 `standard_seconds`、`theoretical_seconds` 保持统一口径
- 金额与费率建议统一使用 `numeric`，避免前端计算和浮点误差
- `remark` 允许为空，不强制默认空字符串

### 3.3 自动计算字段

建议新增以下自动计算字段：

| 字段名 | 类型建议 | 公式 |
| --- | --- | --- |
| labor_cost | numeric(12,4) | `standard_seconds / 3600 * labor_rate` |
| equipment_cost | numeric(12,4) | `equipment_rate * theoretical_seconds / 3600` |
| tooling_consumable_cost | numeric(12,4) | `tool_rate + cutting_fluid_rate + fixture_rate` |
| inspection_cost | numeric(12,4) | `labor_rate * inspection_seconds / 3600` |
| overhead_cost | numeric(12,4) | `daily_management_cost * standard_seconds / 3600 / daily_total_hours` |
| total_cost | numeric(12,4) | 上述各项合计 |

### 3.4 计算实现建议

推荐优先采用“数据库生成列”或数据库统一计算表达式，而不是前端手算。

原因：

- 所有入口看到的成本结果一致
- Excel 导出可直接复用数据库返回值
- 避免前端、服务层、导出工具三处各写一套公式
- 后续如果接入报表或筛选，扩展成本更低

对除零场景必须兜底：

- 当 `daily_total_hours <= 0` 时，`overhead_cost = 0`
- 所有费率/工时为空时按 `0` 计算

### 3.5 重要设计提醒

`daily_management_cost` 和 `daily_total_hours` 从业务语义上更像“日级公共参数”，并不是某个工序/型号的固有属性。

V1 为了快速交付，可以先按你的要求直接放在 `process_standards` 每一行上。

但这会带来两个问题：

- 同一天同一批数据可能需要重复维护很多次
- 后期如果要统一调价，会出现批量改多行的问题

因此建议在计划评审时确认：

- 如果追求最短交付周期，V1 直接加在 `process_standards`
- 如果更看重长期可维护性，V2 可再拆出独立“成本参数表”

## 4. 前端改造方案

### 4.1 页面定位

建议继续复用现有 StandardTimeList 模块，做“能力升级”而不是完全重写：

- 路由短期保持 `standard-time-list` 不变，降低联动改造范围
- 菜单、页头、弹窗标题、提示语统一改为“成本核算”
- 代码目录可暂时不改名，先完成功能；若后续要统一命名，再单独做一次目录与符号重构

### 4.2 管理员页面

管理员侧改造点：

1. 搜索区
- 继续保留按型号、工序、更新时间搜索
- 后续如有需要可增加“成本区间”筛选，但不建议首版就做

2. 表格区
- 保留现有基础列：型号、工序、标准工时、理论工时
- 新增人工维护字段列
- 新增自动计算字段列
- 建议按“基础信息 / 费率参数 / 自动成本”分组展示，避免列过宽不可读

3. 表单区
- 基础信息：型号、工序、标准工时、理论工时
- 手工输入区：人工费率、设备费率、刀具费率、切削液费率、工装费率、检验工时、日管理总费用、日总工时、备注
- 自动计算区：只读显示人工成本、设备成本、刀具辅料成本、检验成本、单品分摊额、合计

4. 交互要求
- 新建/编辑时自动计算字段实时展示，但提交时不作为可编辑值写入
- 删除逻辑保持现状，继续受外键约束保护
- 继续使用管理员批量勾选能力，为导出选中数据提供入口

### 4.3 班组长页面

建议首版保持现有边界：

- 班组长继续只维护 `theoretical_seconds`
- 不开放费率和成本字段编辑
- 手机端仍显示“理论工时”视图即可

原因：

- 现有权限模型明确区分管理员与班组长职责
- 成本参数通常属于管理口径，不适合下放到班组长端
- 可以避免本次改造顺带扩大 RLS 和前端权限面

### 4.4 导出选中数据

建议新增一套专用 Excel 导出工具，例如：

- `src/utils/costAccountingExcel.ts`

导出内容建议包括：

- 型号、工序
- 标准工时、理论工时、检验工时
- 各项费率
- 人工成本、设备成本、刀具辅料成本、检验成本、单品分摊额、合计
- 备注、更新时间

交互建议：

- 仅管理员可见导出按钮
- 未选中数据时按钮禁用
- 首版按“当前页已勾选数据”导出，和现有表格选择能力保持一致

如果你需要“跨页选中后统一导出”，则要额外改造选中态管理，这不建议混入首版范围。

## 9. 风险与关注点

1. 命名风险

- 当前菜单已写“成本核算”，但页头和功能提示仍是“标准工时”
- 若本次只改部分文案，用户会感知混乱

2. 数据语义风险

- `daily_management_cost` 与 `daily_total_hours` 不是稳定的工序属性，后期可能需要抽离成独立表

3. 精度风险

- 金额字段必须统一使用 `numeric`
- 前端显示时要控制小数位，但数据库存储不要过早截断

4. 范围膨胀风险

- 如果本次同时做跨页导出、汇总报表、统计分析，周期会明显拉长
- 建议首版只做单条成本核算 CRUD + 选中导出

## 10. V1 定版方案

本次按以下 4 项决策直接落地，不再作为前置待确认项：

1. `daily_management_cost` 和 `daily_total_hours` 首版直接存到每条 `process_standards` 记录上

- 这样可以最小化数据库改造范围，直接复用现有 `process_standards` CRUD 页面
- 这不是最终最优的数据建模，但对于首版交付是最稳妥方案
- 后续如果发现日级参数复用压力较大，再拆分独立“成本参数表”作为 V2 演进

2. 导出范围首版只支持“当前页勾选数据导出”

- 直接复用现有表格勾选状态，不引入跨页选择缓存逻辑
- 实现成本低，用户心智清晰，回归范围更小
- 如后续确认存在跨页导出刚需，再单独增加“全局勾选导出”能力

3. `刀具费率 / 切削液费率 / 工装费率` 首版按“每支固定金额”处理

- 也就是按你当前描述，直接视为单件固定成本输入项
- 自动计算时三者直接求和，得到“刀具辅料成本（合计）”
- 首版不引入按寿命、按用量、按加工节拍折算的复杂模型

4. 页面对外统一改名为“成本核算”，但路由继续保留 `standard-time-list`

- 菜单、页头、页面标题、弹窗标题、按钮文案全部统一为“成本核算”
- 路由保留现状，避免扩大联动改造范围
- 代码目录和组件名首版可暂不重命名，避免把业务改造和重构混在一起

### 10.1 按该方案落地后的影响

这意味着本次开发边界已经明确：

- 数据主表仍为 `process_standards`
- 首版不做跨页导出
- 首版不做独立成本参数中心
- 首版不做复杂刀具寿命分摊模型
- 首版只统一用户可见文案，不强制统一内部目录命名

### 10.2 版本边界说明

以下内容明确不放入本次 V1：

- 跨页累计勾选导出
- 日级管理费用独立配置表
- 多版本成本参数模板
- 基于刀具寿命、液体用量、工装摊销周期的高级成本模型
- 全量目录、符号、路由重命名

## 11. 推荐实施顺序

建议按下面顺序推进：

1. 先按 V1 定版方案做数据库 migration 和类型刷新
2. 再做服务层和管理员表单/表格
3. 然后接入当前页勾选导出
4. 最后做回归验证和文案统一
