# 物料转移单开发计划

## 1. 背景与目标

为车间管理新增一个“物料转移单”功能，支持桌面端完整 CRUD，用于记录物料在不同接收车间之间的转移情况。

本次版本目标：

- 新增一张转移单数据表
- 新增一个桌面端页面及路由入口
- 参考生产工单页面的交互方式
- 完成列表、新增、编辑、删除的完整闭环

本次版本范围限定：

- 先做桌面端
- 优先支持管理员使用
- 暂不处理员工移动端专属页面

## 2. 需求摘要

### 2.1 表单字段

转移单包含以下字段：

- 项目号：支持按项目号查询
- 物料信息：根据项目号自动带出
- 转移数量：整数类型，必须大于 0
- 操作人：关联员工表
- 接收车间：前端下拉选择
- 接收人：手工输入

接收车间固定选项：

- 挤压
- 时效
- 大氧化
- 小氧化
- 精切
- 精加工
- 喷涂
- 抛光
- 包装
- 仓库

### 2.2 页面要求

- 新增独立路由
- 页面交互参考生产工单页面
- 支持桌面端 CRUD

## 4. 数据库设计方案

### 4.1 表名建议

建议表名：material_transfers

### 4.2 字段设计

建议字段如下：

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| id | uuid | 主键 |
| project_no | text | 项目号 |
| product_model | text | 型号快照 |
| length_mm | numeric | 长度快照 |
| customer_model | text | 客户型号快照 |
| transfer_quantity | integer | 转移数量 |
| operator_employee_id | uuid | 操作人，关联 employees.id |
| target_workshop | text | 接收车间 |
| recipient_name | text | 接收人 |
| remark | text | 备注，可选 |
| created_at | timestamptz | 创建时间 |
| updated_at | timestamptz | 更新时间 |

### 4.3 为什么建议保留物料快照字段

虽然物料信息可以由项目号实时查出，但建议在转移单表中同时存储：

- product_model
- length_mm
- customer_model

原因：

- 避免源数据变化导致历史记录显示漂移
- 列表页展示时减少二次查询
- 导出和审计时更稳定

### 4.4 约束与索引

建议新增以下约束：

- transfer_quantity > 0
- target_workshop 必须属于预设枚举值
- operator_employee_id 外键关联 employees.id

建议新增以下索引：

- project_no
- operator_employee_id
- target_workshop
- created_at desc

### 4.5 RLS 策略

由于当前系统已对员工体系启用严格 RLS，本功能不能只建表不配策略。

第一版建议：

- admin：允许全量查询、创建、更新、删除
- employee：暂不开放

后续若扩展员工侧录入，再增加：

- employee 仅能查看和维护自己的记录

## 6. 项目号与物料信息实现方案

项目号选择建议直接参考生产工单现有实现。

可复用能力：

- 项目号 options 构建逻辑
- 项目号下拉搜索逻辑
- 项目号下拉渲染逻辑
- 销售订单项目号查询 hook

第一版“物料信息”建议定义为以下三个字段：

- 型号 product_model
- 长度 length_mm
- 客户型号 customer_model

如果业务后续需要展示更完整的物料明细，再单独扩展。

## 9. 验收标准

满足以下条件视为第一版完成：

- 管理员能从菜单进入物料转移单页面
- 可以新增一条转移记录
- 项目号选择后能自动带出物料信息
- 可以编辑已有记录
- 可以单条删除和批量删除
- 列表能展示项目号、物料信息、转移数量、操作人、接收车间、接收人
- 非法数量无法提交
- 构建通过
- 不影响现有生产工单功能

## 10. 风险与注意事项

### 10.1 物料信息定义要先收口

当前代码里能稳定复用的“物料信息”主要是：

- project_no
- product_model
- length_mm
- customer_model

如果业务想表达的是更完整的 BOM 或原材料明细，需要另补来源设计。

### 10.2 权限不能后补

当前生产相关功能已经使用严格员工隔离 RLS，新表如果不同时加 policy，很容易在联调时出现读不到或写不进去的问题。

### 10.3 员工端暂不纳入第一版

当前应用中 employee 角色默认走移动端布局。既然本次明确“先做桌面端”，那么第一版更合理的策略是只做 admin 闭环。
