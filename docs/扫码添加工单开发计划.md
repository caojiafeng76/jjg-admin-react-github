# 扫码添加工单开发计划

## 1. 需求背景

当前员工端已经具备以下基础能力：

- 移动端统一容器与底部导航
- 我的工单页面与员工视角权限控制
- 生产工单主表、工序明细表、工单明细编辑能力
- 项目号扫码识别与项目号关联信息自动带出能力

本次需求不是新增一套独立工单体系，而是在现有生产工单能力上补齐“移动端扫码快速录入”闭环，重点解决以下问题：

- 员工手机端缺少直接进入扫码录入的快捷入口
- 当前“添加工序明细”仍以弹窗为主，不适合手机端扫码场景
- 当前员工端同一天已有工单时会阻止再次创建，无法直接把新扫码工序追加到当天工单
- 下拉选择仍偏桌面交互，需要改成底部弹出式选择体验

## 2. 本次目标

### 2.1 业务目标

- 员工可在移动端通过扫码快速进入“新增工序明细”流程
- 扫码后自动填充项目号，并带出型号、长度、客户型号
- 员工可手动改项目号，系统重新带出对应关联信息
- 提交时按“当前登录人 + 当前日期”判断：
  - 当天已有工单：直接把本次工序明细追加到该工单
  - 当天无工单：先创建主工单，再写入工序明细

### 2.2 交互目标

- 移动端新增一个“扫码工单”快捷入口
- 录入表单由弹窗改为独立页面
- 项目号、工序、机器编号等选择在移动端以底部弹层方式展示
- 页面尽量减少输入步骤，扫码后优先自动带值

## 3. 现状复用分析

基于现有代码，以下能力可以直接复用：

### 3.1 移动端容器

- 员工端移动容器已经存在，入口位于 EmployeeMobileLayout
- 当前底部导航已有“我的工单”等入口，可在同一区域补充扫码录入入口

### 3.2 工单主流程

- production-order 模块已经区分管理员视角与员工视角
- 已存在 useProductionOrders、useProductionOrder、useProductionOrderItems 等查询与提交能力
- 已存在 createProductionOrder、updateProductionOrder、addProductionOrderItem 等服务函数

### 3.3 当天工单检查

- 现有 production-order 页面已经存在“当前员工当天是否已有工单”的检查逻辑
- 当前逻辑用于阻止员工一天创建多张工单
- 本次只需要把这段“阻止创建”的逻辑改造成“存在则复用该工单继续追加明细”

### 3.4 项目号扫码与自动带值

- ProductionOrderItemForm 已经集成 ProjectNoScanButton
- ProjectNoScanButton 已具备以下能力：
  - 扫描二维码
  - 识别项目号或订单二维码
  - 解析项目号对应的型号、长度、客户型号等信息
- 项目号切换后，现有逻辑已经能自动刷新工序候选与标准工时

结论：本次应优先采用“页面抽离 + 交互调整 + 提交流程改造”的方案，而不是重写服务层。

## 4. 页面方案

## 4.1 新增入口

建议在员工端移动容器中新增一个“扫码工单”入口，样式为：

- 上方扫码图标
- 下方文案“工单”

推荐放置位置：

- 优先放在员工端底部导航，与“我的工单”同级
- 若底部导航不希望增加路由数量，可在“我的工单”页头增加一个醒目的扫码快捷卡片

推荐方案：底部导航新增独立入口。

原因：

- 点击路径最短
- 与移动端高频录入场景一致
- 更容易形成“扫码即录入”的固定心智

### 4.2 新增页面

把当前“添加工序明细”弹窗拆为独立移动端页面，例如：

- 路由建议：/production-order/scan
- 页面命名建议：MobileProductionOrderScanPage 或 ScanProductionOrderItemPage

页面职责：

- 承载扫码录入流程
- 展示工序明细表单
- 负责按当天工单自动归并提交

不建议继续使用 Modal 直接承载，原因如下：

- 移动端弹窗内再放长表单，滚动和键盘交互体验差
- 底部弹层式选择与页面级导航更容易组合
- 扫码失败、重扫、回退等场景在独立页面里更清晰

## 5. 页面交互流程

### 5.1 进入页面

员工点击“扫码工单”入口后进入扫码录入页。

页面初始态建议包含：

- 顶部标题：扫码添加工单
- 扫码主按钮
- 最近一次识别结果提示区
- 工序明细表单区
- 底部固定提交按钮

### 5.2 第一步：扫码

用户点击扫码后：

1. 调起现有扫码能力
2. 识别二维码中的项目号或订单二维码
3. 自动回填以下字段：
   - 项目号
   - 型号
   - 长度(mm)
   - 客户型号
4. 根据型号、长度、料号加载工序候选

### 5.3 第二步：允许人工修正项目号

虽然扫码默认带值，但项目号字段必须允许人工修改。

修改项目号后应自动触发：

- 重新查询项目号关联信息
- 清空并重置当前工序、标准工时、机器编号
- 重新加载新的工序候选列表

### 5.4 第三步：底部弹层选择

移动端下列选择项统一改为底部弹出：

- 项目号选择
- 工序选择
- 机器编号选择

交互建议：

- 点击输入框后，从页面底部弹出 Drawer / Action Sheet 风格面板
- 面板顶部保留标题和关闭按钮
- 面板内部支持搜索、滚动和点击即回填
- 回填后自动关闭面板

这样可以对齐参考图中的“从底部弹出”交互，不再依赖桌面式下拉浮层。

### 5.5 第四步：填写数量类字段

保留当前工序明细表单中的核心字段：

- 来料接收数
- 成品合格数
- 加工不良数量
- 原料不良数量
- 外协不良数
- 调机不良
- 备注

移动端第一页建议先保留必要字段，不新增额外复杂逻辑。

## 6. 提交流程改造

## 6.1 当前问题

现有员工端逻辑在新建工单时会判断“该员工当天是否已有工单”。如果已有，则直接报错并阻止创建。

这与本次需求冲突。

## 6.2 目标流程

提交扫码工序明细时，系统按以下顺序处理：

1. 获取当前登录员工信息
2. 取当前业务日期
3. 查询该员工在该日期是否已有 production_orders 记录
4. 如果已有：
   - 取该工单 id
   - 直接新增一条 production_order_items
5. 如果没有：
   - 先创建 production_orders 主表记录
   - 再新增 production_order_items

## 6.3 推荐实现方式

建议新增一个明确的服务方法，例如：

- getEmployeeOrderByDate(employeeId, orderDate)

该方法职责只做一件事：

- 返回指定员工在指定日期的工单
- 若不存在则返回 null

移动端扫码页提交时统一走：

- ensureTodayOrderThenAddItem

建议把流程封装成单独的提交编排函数，而不要散落在页面事件内部，避免后续“手动添加工序”“继续录入下一条”场景重复写逻辑。

## 6.4 默认值策略

若当天不存在工单，需要自动创建主工单，建议默认值如下：

- order_date：当天日期
- employee_id：当前登录员工 id
- shift：沿用现有默认逻辑，未识别时默认白班
- work_hours：若当前页面无工时录入，则先使用系统默认值或允许后续补录
- remark：可空

这里需要在开发前确认一个业务点：

- 扫码页是否同时录入出勤工时
- 若不录入，系统是否允许创建 work_hours 为 0 的工单

从现有服务校验看，work_hours 不能小于 0，因此技术上允许先创建 0 工时工单，再由员工后续补录。

## 7. 组件拆分建议

### 7.1 建议新增文件

建议在现有 production-order 模块内新增移动端扫码录入页面及辅助组件，而不是新开完全独立 feature。

建议结构：

```text
src/features/production-order/
  MobileProductionOrderScanPage.tsx
  MobileBottomSelectSheet.tsx
  useEmployeeTodayProductionOrder.ts
  useSubmitScannedProductionOrderItem.ts
```

### 7.2 建议复用/抽离的现有逻辑

- 复用 ProjectNoScanButton
- 复用 useSalesOrdersProjectNos
- 复用 useOperationsByModel
- 复用 useMachineEquipmentOptions
- 复用 addProductionOrderItem
- 复用 createProductionOrder

### 7.3 建议从现有弹窗表单中抽离的部分

当前 ProductionOrderItemForm 中可抽离出以下通用逻辑：

- 项目号回填逻辑
- 扫码结果合并逻辑
- 工序候选加载逻辑
- 机器编号候选过滤逻辑
- 提交前的数量校验逻辑

建议把“字段逻辑”与“弹窗壳子”拆开，形成：

- 共享表单体
- 桌面端继续放在 Modal 中
- 移动端放到独立页面中

这样可以避免未来改一个字段，桌面端和移动端各修一份。

## 8. 路由与导航调整建议

### 8.1 路由

建议新增移动端扫码录入路由，并继续挂在员工侧权限范围内。

建议示例：

- /production-order/scan

### 8.2 导航

需要同步调整：

- 员工端移动底部导航
- 页头标题映射
- 可能的返回路径与面包屑文案

### 8.3 页面返回逻辑

扫码页提交成功后建议：

- 返回“我的工单”页面
- 默认定位到当天工单或直接进入当天工单详情

推荐方案：

- 提交成功后直接进入当天工单详情

原因：

- 员工通常还会继续补录第二条明细
- 可以立刻看到本次录入结果

## 9. 数据与权限要求

本次不新增角色模型，继续沿用当前 employee / team_leader 员工侧角色规则。

需要保证：

- 扫码页只能由员工侧角色进入
- 创建或复用工单时 employee_id 必须强制取当前登录人
- 不允许前端提交其他员工 id 来写入工单

如果后续要把“当天工单自动归并”放到数据库函数或 RPC 中处理，需同步评估 RLS 与服务端校验；本期先用前端编排 + 现有服务层即可。

## 10. 风险与注意点

### 10.1 重复提交风险

扫码页提交按钮需防重入，避免连续点击导致同一明细重复写入。

### 10.2 当天工单并发创建

如果员工在两台设备同时提交，存在极低概率出现当天重复工单。

一期可先通过前端提交态与再次查询降低概率；若后续出现实际问题，再考虑：

- 数据库唯一约束
- RPC 原子化创建/复用

### 10.3 移动端选择组件体验

如果继续直接使用 Select 默认弹出层，移动端体验会明显偏桌面。

本次必须明确改成底部弹层，不建议只改样式。

### 10.4 工时字段业务确认

需要在开发前确认扫码页是否同步维护出勤工时，否则提交创建工单时默认值策略要先定下来。

## 11. 验收标准

满足以下条件即认为本次需求完成：

- 员工端可见“扫码工单”入口
- 点击入口后进入独立页面，不再使用旧弹窗完成扫码录入
- 扫码后项目号自动填充，并自动带出型号、长度、客户型号
- 项目号允许人工修改，修改后相关字段重新刷新
- 项目号、工序、机器编号在移动端以底部弹层方式选择
- 当天已有工单时，提交后把新工序明细追加到当天工单
- 当天没有工单时，提交后自动创建工单并写入工序明细
- 提交成功后页面反馈明确，且可以看到新增结果
- lint 与 build 通过

## 12. 建议开发顺序

### 第一阶段：方案落地

- 明确扫码页的路由、入口位置和返回路径
- 确认扫码页是否需要录入出勤工时
- 确认提交成功后跳列表还是跳详情

### 第二阶段：基础能力抽离

- 抽离 ProductionOrderItemForm 中可共享的字段逻辑
- 新增员工当天工单查询 hook / helper
- 新增“确保当天工单存在后再新增明细”的提交流程函数

### 第三阶段：页面与交互

- 新增扫码录入页
- 接入扫码能力
- 将选择控件改为底部弹层
- 新增移动端入口

### 第四阶段：联调与回归

- 验证有工单、无工单两条主路径
- 验证扫码成功、扫码失败、人工改项目号、重复提交等边界场景
- 执行 lint / build 回归
