# 员工手机端工单录入开发计划

## 1. 背景与现状

当前系统已经具备以下基础能力：

- 已接入 Supabase Auth，支持邮箱密码登录
- 已有员工表 `employees`
- 已有生产工单主表 `production_orders`，并通过 `employee_id` 关联员工
- 已有生产工单明细 `production_order_items`
- 已有后台管理端的工单列表、详情、增删改能力

当前仍缺少以下关键能力：

- 员工账号与员工资料没有建立一一绑定关系
- 路由层只有“是否登录”，没有“角色”和“数据归属”控制
- 数据访问仍以普通前端查询为主，没有数据库级隔离策略
- 现有工单页面偏桌面后台，不适合手机端高频录入

因此，本次改造的重点不是简单做一个手机页面，而是补齐“员工身份识别 + 数据权限隔离 + 移动端录入体验”三层能力。

---

## 2. 建设目标

### 2.1 业务目标

- 员工可通过手机登录系统并录入自己的工单数据
- 员工只能查看自己的工单和相关明细
- 员工只能编辑自己创建或归属自己的工单数据
- 管理员仍可查看和管理全部数据

### 2.2 技术目标

- 以当前 React + Vite + Supabase 架构为基础完成改造，不额外拆分独立 App
- 优先采用 H5 移动端方案，确保微信内置浏览器和常见安卓手机可用
- 将权限控制下沉到 Supabase RLS，而不是只依赖前端隐藏按钮
- 复用现有工单、员工、标准工时、项目号等数据服务，减少重复建设

---

## 3. 推荐方案

推荐采用“单系统双入口”的方案：

- 后台入口：保留现有管理端，用于管理员维护员工、标准工时、全量工单
- 手机入口：新增员工移动端页面，只提供员工本人常用能力

### 3.1 为什么不建议直接复用现有工单页面

现有生产工单页面以表格、弹窗、批量勾选、导出为主，明显偏桌面后台操作模型。直接在手机上复用会带来以下问题：

- 表格横向信息过多，手机上难以浏览
- 弹窗内再嵌表单和明细编辑，操作复杂
- 批量选择、导出、管理操作对员工无意义
- 很难做好“只能看自己”的心智边界

因此建议新增一套移动端页面，保留后台页面不动。

---

## 4. 权限模型设计

### 4.1 角色划分

建议至少区分两类角色：

- 管理员：查看和维护所有员工、工单、标准工时、报表
- 员工：仅查看和编辑自己的工单数据

如后续需要，可继续扩展为：

- 班组长：查看本组员工数据
- 审核员：可审核但不可修改员工已提交数据

### 4.2 员工与账号绑定

建议新增账号绑定关系，不要只靠员工姓名匹配。

推荐两种实现方式，优先选择方案 A：

#### 方案 A：扩展 employees 表

在 `employees` 表中新增字段：

- `auth_user_id uuid unique null`：绑定 Supabase Auth 用户 ID
- `role text not null default 'employee'`：角色
- `is_active boolean not null default true`：是否启用
- `mobile varchar null`：手机号，可选

优点：

- 结构简单，改造成本低
- 与当前 `production_orders.employee_id` 直接兼容
- 对现有前端改动最小

#### 方案 B：新增 user_profiles 表

单独建立用户档案表，关联 Auth 用户与员工资料。

优点：扩展性更好。

缺点：当前项目规模下会增加联表与维护复杂度。

结论：当前阶段建议先用方案 A。

## 5. 数据权限设计

### 5.1 核心原则

权限必须以后端数据库规则为准，前端只做体验层限制。

也就是说，即使员工篡改前端请求，也不能读取或修改别人的工单。

### 5.2 RLS 设计建议

对以下表开启 RLS：

- `employees`
- `production_orders`
- `production_order_items`

### 5.3 权限规则

#### employees

- 员工只能读取自己的员工资料
- 管理员可以读取所有员工资料
- 普通员工不能直接修改员工主数据，避免自行改姓名、角色、绑定关系

#### production_orders

- 员工只能查询 `employee_id = 当前登录员工ID` 的工单
- 员工只能插入 `employee_id = 当前登录员工ID` 的工单
- 员工只能更新 `employee_id = 当前登录员工ID` 的工单
- 员工不能删除他人数据
- 管理员拥有全量权限

#### production_order_items

- 员工只能查看其本人所属工单下的明细
- 员工只能增删改其本人所属工单下的明细
- 管理员拥有全量权限

### 5.4 关键控制点

为避免员工伪造 `employee_id`，建议：

- 插入工单时后端自动校验当前用户对应的员工 ID
- 员工端表单中不提供“选择员工”字段
- 服务层对员工端请求再次做参数兜底，写入时强制覆盖为当前员工 ID

## 6. 业务流程设计

### 6.1 管理员流程

1. 创建员工资料
2. 为员工创建 Supabase Auth 账号
3. 将 Auth 用户绑定到 `employees.auth_user_id`
4. 设置角色为 `employee`
5. 将登录方式发给员工

### 6.2 员工流程

1. 手机访问员工端入口
2. 使用账号密码登录
3. 进入“我的工单”列表
4. 新建工单
5. 添加工序明细
6. 查看历史工单
7. 编辑自己的工单

### 6.3 可选流程增强

为了控制数据质量，建议预留“提交状态”：

- 草稿：员工可持续编辑
- 已提交：员工不可再改，等待管理端审核
- 已退回：员工可重新修改

如果当前先求快速上线，可第一期先不上审核流，只做“本人可编辑”。

## 7. 前端改造方案

### 7.1 总体策略

在现有 React 项目内新增移动端页面，不单独拆仓库。

建议新增两类路由：

- 管理端路由：保持现状
- 员工端路由：如 `/m/login`、`/m/orders`、`/m/orders/:id`

### 7.2 建议新增模块

建议新增一个独立 feature：

- `src/features/mobile-production-order/`

建议页面结构：

- `MobileLogin`：手机登录页，可复用现有认证能力
- `MyProductionOrderList`：我的工单列表
- `MyProductionOrderForm`：新建/编辑我的工单
- `MyProductionOrderDetail`：查看工单详情
- `MyProductionOrderItemForm`：录入工序明细

### 7.3 手机端交互原则

- 一屏只做一件事
- 卡片式列表替代表格
- 底部主按钮固定，如“新建工单”“保存”
- 减少字段数量，默认值尽量自动带出
- 工序明细使用分步录入，而不是桌面端的大型弹窗

### 7.4 员工端页面建议

#### 我的工单列表

显示字段建议：

- 日期
- 合格工时
- 效率
- 备注摘要
- 状态

支持操作：

- 新建
- 查看
- 编辑本人数据

不提供：

- 导出
- 批量删除
- 查看他人工单
- 员工切换筛选

#### 工单编辑页

主表字段建议保留：

- 日期
- 出勤工时
- 备注

不显示：

- 员工选择

由系统自动确定：

- `employee_id`

#### 工序明细编辑

保留核心字段：

- 项目号
- 型号
- 工序
- 标准工时
- 合格数量
- 不良原因与数量
- 奖励秒数

建议优化：

- 常用不良原因预设快捷项
- 输入完成后自动回到明细列表
- 支持复制上一条明细，提高连续录入效率

## 8. 服务层改造方案

### 8.1 当前问题

现有服务方法主要面向后台全量管理场景，例如：

- 查询列表时支持按员工筛选
- 创建工单时允许传入 `employee_id`
- 更新工单时未区分管理员和员工上下文

这对员工端不够安全，也不够清晰。

### 8.2 建议做法

新增一组“员工自助接口封装”，与后台接口分离：

- `getMyEmployeeProfile()`
- `getMyProductionOrders()`
- `getMyProductionOrderById(id)`
- `createMyProductionOrder(values)`
- `updateMyProductionOrder(id, values)`
- `addMyProductionOrderItem(values)`
- `updateMyProductionOrderItem(id, values)`
- `deleteMyProductionOrderItem(ids)`

这组方法的原则是：

- 不接受外部传入 `employee_id`
- 内部基于当前登录用户反查绑定员工
- 所有查询只返回本人数据

这样可以让移动端代码语义更清楚，也能降低误用后台接口的风险。

## 9. 管理端改造方案

为了支持员工端上线，后台至少需要补两块能力。

### 9.1 员工管理页增强

在现有员工管理中增加：

- 账号绑定状态
- 角色
- 启用状态
- 重置密码入口或重新发送登录方式说明

建议新增操作：

- 绑定账号
- 解除绑定
- 禁用员工登录

### 9.2 菜单与权限控制

当前菜单是全量展示，后续需要按角色裁剪。

建议：

- 管理员登录后仍显示完整后台菜单
- 员工登录后默认跳转移动端首页，不显示后台管理菜单

建议新增迁移，主要包含以下内容：

1. 为 `employees` 增加 `auth_user_id`
2. 为 `employees` 增加 `role`

## 13. 风险与应对

### 风险 1：只做前端限制，不做数据库隔离

后果：员工可通过抓包越权读取或修改他人数据。

应对：必须上 RLS，这是本项目的硬要求。

### 风险 2：员工账号与员工资料绑定不稳定

后果：出现串号、数据归属错误。

应对：使用 `auth_user_id` 作为唯一绑定键，不使用姓名匹配。

### 风险 3：直接复用后台页面导致手机体验差

后果：录入效率低，员工不愿意使用。

应对：单独做移动端页面，简化流程。

### 风险 4：历史数据编辑边界不清晰

后果：员工可修改很久以前的数据，影响报表稳定性。

应对：建议明确编辑窗口，例如仅允许编辑最近 3 天或未提交数据。

## 14. 第一阶段建议范围

建议先做 MVP，范围控制如下：

- 员工账号登录
- 员工与账号绑定
- 员工只能查看自己的工单
- 员工只能新增和编辑自己的工单及明细
- 管理员仍使用现有后台
- 员工端提供基础手机页面

先不做：

- 审核流
- 班组长层级权限
- 消息通知
- 小程序
- 离线缓存

这样能最快验证员工自助录入是否能跑通。

## 16. 开发任务清单

这一节用于直接排期和分工。建议按“数据库 / 前端基础 / 员工端页面 / 管理端增强 / 测试上线”五个工作包推进。

### 16.1 数据库与权限

#### 工作包 A：员工账号绑定基础

目标：让登录账号与员工资料建立唯一映射。

任务清单：

- 为 `employees` 增加 `auth_user_id`
- 为 `employees` 增加 `role`
- 为 `employees` 增加 `is_active`
- 为 `employees.auth_user_id` 建唯一索引
- 补一条管理员初始化方案，明确首个管理员如何创建
- 明确旧员工数据的补录策略

验收标准：

- 每个员工最多绑定一个登录账号
- 员工可通过账号反查到唯一员工资料
- 未绑定账号的员工不能直接登录员工端

建议负责人：

- 后端 / 数据库

#### 工作包 B：RLS 权限策略

目标：从数据库层面限制员工只能访问自己的数据。

任务清单：

- 为 `employees` 开启 RLS
- 为 `production_orders` 开启 RLS
- 为 `production_order_items` 开启 RLS
- 编写管理员全量访问策略
- 编写员工仅访问本人数据策略
- 验证插入、更新、查询、删除四类操作权限
- 明确员工禁改字段范围

验收标准：

- 员工账号无法查询他人工单
- 员工账号无法更新他人工单和明细
- 员工账号即使手工构造请求也无法写入他人 `employee_id`
- 管理员账号仍可正常访问全量数据

建议负责人：

- 后端 / 数据库

#### 工作包 C：可选状态字段

目标：为后续审核流预留能力。

任务清单：

- 评估是否新增 `production_orders.status`
- 评估是否新增 `submitted_at`
- 如做审核流，再补充状态流转约束

验收标准：

- 第一阶段不做审核流时，不影响主链路上线
- 后续增加审核流时无需推翻当前结构

建议负责人：

- 产品 + 后端

### 16.2 前端基础能力

#### 工作包 D：登录上下文增强

目标：让前端知道“当前是谁、是什么角色、绑定了哪个员工”。

任务清单：

- 扩展 `AuthContext`，增加当前角色信息
- 增加当前员工资料查询 hook
- 增加“未绑定员工资料”的异常处理界面

验收标准：

- 登录后可区分管理员和员工
- 员工登录后可拿到自己的员工资料

建议负责人：

- 前端

目标：让管理员和员工进入不同界面。

## 18. 更细的实施顺序

这一节不是按模块分工，而是按真实开发依赖顺序编排。这样做的原因很简单：有些工作包可以并行，但有些必须先完成，否则后面的开发会反复返工。

### 18.1 第一步：先定死 4 个业务口径

这一步不写代码，但必须在开发前确认。建议当天就确认完。

需要确认的 4 个问题：

1. 员工使用什么方式登录
2. 员工能否修改历史工单，允许修改几天内的数据
3. 第一阶段是否存在“提交后不可修改”
4. 管理员如何创建和绑定员工账号

如果这一步不先确认，后面最容易返工的是：

- 数据库字段设计
- RLS 策略
- 页面按钮文案
- 编辑权限边界

输出物：

- 一份确认记录，哪怕只是一段结论文字也可以

### 18.2 第二步：先做数据库，再做前端

这是整个项目最关键的顺序。不要反过来。

建议顺序：

1. 改表结构
2. 做账号绑定
3. 开 RLS
4. 验证权限
5. 再开始前端页面开发

原因：

- 员工端的所有页面都依赖“当前登录用户能拿到自己的员工身份”
- 如果数据库和 RLS 没先定好，前端开发出来也只是临时假数据流程
- 权限边界不稳定时，前端接口层会不断改签名

### 18.3 第三步：完成登录上下文和路由分流

数据库能力稳定后，第一批前端代码应该是基础设施，不是页面。

优先顺序：

1. 扩展 `AuthContext`
2. 增加当前员工资料查询 hook
3. 增加角色识别逻辑
4. 完成 `/m/*` 路由分流
5. 完成员工与管理员登录后的默认跳转

只有这一步完成后，员工端页面开发才真正有稳定承载点。

### 18.4 第四步：先打通“我的工单主流程”

主流程优先级建议如下：

1. 我的工单列表
2. 新建工单主信息
3. 进入工单详情
4. 新增一条工序明细
5. 编辑工序明细

这一阶段先不追求体验极致，先追求闭环可用。也就是至少能完成下面这个链路：

员工登录 -> 进入我的工单 -> 新建工单 -> 添加明细 -> 返回列表 -> 再次打开并继续编辑

如果这个闭环没打通，就不要过早做细节美化。

### 18.5 第五步：再补体验优化和管理端增强

主流程跑通后，再做下面这些：

- 明细录入快捷项
- 复制上一条明细
- 离开未保存提示
- 底部安全区和软键盘适配
- 员工管理页的绑定状态、角色、启用状态
- 后台菜单裁剪

这些都重要，但排在主链路之后更合理。

### 18.6 第六步：最后做真机验证和灰度试点

最后一步不要只在桌面浏览器里验证。必须真机试。

建议顺序：

1. 开发环境手机访问验证
2. Android Chrome 验证
3. 微信浏览器验证
4. 2 到 5 名员工灰度试点
5. 收集问题并修一轮

如果在试点前就全量上线，通常会在以下地方出问题：

- 键盘遮挡按钮
- 表单项太多
- 员工看不懂字段
- 连续录入效率过低

## 21. 每天结束时的检查点

为了避免开发过程看起来很忙，但实际没有形成闭环，建议每天结束都检查下面 4 个问题：

1. 今天完成的内容，是否真的可演示
2. 今天完成的内容，是否已经接到真实数据而不是假数据
3. 今天新增的逻辑，是否考虑了员工只能访问本人数据
4. 今天的改动，是否影响了现有后台能力

如果这 4 个问题有两个以上回答为“否”，说明当天虽然写了代码，但没有真正推进上线结果，需要调整顺序。
