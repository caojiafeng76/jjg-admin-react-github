## Context

`production-scheduling` 当前在菜单和 Header 中显示为“排产计划”，页面由 `ComingSoonPage` 占位。订单管理、生产工单、成本核算和岗位基础数值已经分别有服务层与 feature 页面，但缺少一个按订单聚合生产推进情况的入口。

## Goals / Non-Goals

**Goals:**

- 在原排产计划入口提供“订单现状”只读页面。
- 按订单展示计划数量、各成本核算工种岗位合格数量、成品率、完成率和生产状态。
- 支持按订单日期、项目号、型号模糊搜索订单现状。
- 按订单展示物料转移数量、入库数量、接收车间和最近转移记录。
- 支持点击物料转移数量查看该订单的物料转移明细。
- 支持点击岗位产量单元格查看该订单该岗位下不同工序的合格数量汇总和生产工单明细。
- 复用现有 Supabase 服务层、TanStack Query 配置和 Ant Design 表格形态。

**Non-Goals:**

- 不新增或修改数据库结构。
- 不改变订单、生产工单、成本核算、岗位基础数值的写入规则。
- 不在首版实现导出、编辑、打印或复杂排产算法。

## Decisions

- 保留 `production-scheduling` 路由 key，仅把文案和页面能力改为“订单现状”，避免权限、收藏路径和历史入口失效。
- 新增只读汇总服务函数聚合现有数据：订单作为主表，生产工单明细按订单与成本核算工种岗位求和，成本核算 `job_name` 用于生成岗位列候选。
- 岗位匹配以成本核算“工种”字段为唯一岗位来源；匹配时先按型号、工序、料号、长度精确命中，再用去除工序括号后缀后的基础工序做唯一工种兜底，覆盖“冲葫芦孔（上/下）”与生产明细“冲葫芦孔”这类名称差异。
- 搜索条件写入 URL 查询参数，并进入 TanStack Query key；订单日期、项目号和型号筛选在订单读取阶段完成，保证分页总数与筛选结果一致。
- 订单状态页签写入 URL 查询参数，并进入 TanStack Query key；默认显示“生产中”，切换“已结案”时复用同一套搜索条件并重置到第一页。
- 物料转移单通过 `project_no` 与订单关联，按当前页订单批量读取 `material_transfers` 后在前端服务层聚合，避免逐行请求。
- 物料转移明细弹窗复用当前列表查询返回的数据，不额外发起单行明细请求；点击转移数量时展示转移时间、接收车间、转移数量、操作人、接收人和审核状态。
- 岗位产量弹窗复用当前列表查询返回的数据，不额外发起单行明细请求；生产工单明细按订单和工序岗位映射到对应产量单元格，点击时展示该岗位下的工序合格数量汇总和对应明细。
- 首版 UI 使用横向滚动表格，岗位列动态生成，尽量接近用户提供的长图样式，同时保留后台系统现有 Ant Design 操作习惯。
- 生产状态先由完成率和异常/延期信号推导为“正常 / 预警 / 延期”，后续可按用户反馈调整口径。

## Risks / Trade-offs

- [字段口径不完全明确] → 首版以现有订单数量和工单明细数量为统计源，在页面文案中保持状态标签简洁，后续按实际字段反馈调整。
- [转移口径可能细分] → 首版展示全部物料转移单总量，同时单独标出进入“仓库”的入库数量，后续可按车间类别继续拆分。
- [岗位列过多导致横向宽度大] → 使用 Table 横向滚动和紧凑列宽，优先保证信息完整。
- [跨表读取数据量较大] → 首版限制分页，按页面当前订单批次聚合明细；后续如数据量增长再考虑数据库视图或 RPC 汇总。
