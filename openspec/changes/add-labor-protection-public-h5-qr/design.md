## Context

当前劳保领料单功能已经具备后台管理端 CRUD，数据存放在 `labor_protection_data` 与 `labor_protection_requisitions` 两张表中，前端也已有完整的列表、表单和查询 hook。但该能力完全挂在受保护路由下，且数据库 RLS 仅允许管理员访问，因此现场人员无法通过手机直接登记。

本次需求同时涉及前端公开路由、移动端交互、二维码链接生成与打印，以及数据库匿名访问策略，是一个跨模块变更。约束包括：

- 必须继续复用当前 React + Vite + Supabase 架构，不拆独立应用。
- H5 页面需要适配微信内置浏览器与常见手机视口。
- 匿名访问必须只开放最小能力，不能放开后台列表、编辑、删除。
- 二维码海报需要在后台一键打印，且扫码后直达公开登记页。

## Goals / Non-Goals

**Goals:**

- 提供无需登录即可访问的劳保领料登记 H5 页面。
- 允许公开页面读取劳保种类并提交新的领料单记录。
- 在后台领料单页面提供二维码海报打印入口，海报样式适合线下张贴。
- 保持现有后台列表、编辑、删除和查询逻辑不受影响。

**Non-Goals:**

- 不引入新的身份认证、验证码或审核流。
- 不改变劳保领料单现有字段模型。
- 不把后台管理页改造成响应式移动端页面。
- 不实现匿名用户对历史领料单的查询、修改或删除能力。

## Decisions

### 1. 公开登记页使用独立匿名路由，而不是复用后台受保护页面

决策：新增一个挂在 router 顶层的公开路由，使用独立页面组件承载 H5 表单与成功态，不复用现有后台列表页面。

原因：现有后台页面依赖 `ProtectedRoute` 与 `RoleProtectedRoute`，同时交互模型以列表、弹窗和批量操作为主，不适合现场扫码登记。独立匿名页可以避免用户被重定向到登录页，也能按移动端流程单独设计。

备选方案：

- 直接复用后台页面并放开权限：会把管理操作暴露给公开访问者，风险过高。
- 另起独立项目：增加部署与维护成本，不符合当前仓库约束。

### 2. 复用现有表结构，仅新增最小 RLS 策略

决策：继续使用 `labor_protection_data` 作为种类来源，继续使用 `labor_protection_requisitions` 记录登记数据；新增 migration 允许 `anon`/`authenticated` 角色读取劳保种类，并允许公开插入劳保领料单。

原因：现有表结构已经覆盖本次所需字段，前端和后台查询也已完整落地。新增最小策略比引入 RPC 或 Edge Function 更轻量，也更容易与当前 anon key 前端架构对齐。

备选方案：

- 新建公开表单专用表：会造成数据重复与后台统计口径分裂。
- 改用 Edge Function 代写入：安全边界更强，但当前需求仅需最小开放，复杂度不匹配。

### 3. 公共页与后台页复用服务层的输入校验，但分离入口语义

决策：保留现有服务层的字段归一化和校验逻辑，并为公开 H5 页面增加专用 hooks/页面状态；后台管理页面继续使用现有列表与 mutation hooks。

原因：字段校验已经在 service 层收敛，复用可以保证“岗位/领取人去空格、数量为正整数”等规则一致。将公开页面与后台列表拆分，则可以避免 query key、权限语义和交互状态混杂。

备选方案：

- 完全共用后台 hook 和页面：会把列表缓存、弹窗状态和公开页流程耦合在一起。

### 4. 二维码海报使用独立可打印视图模型，二维码值始终由公共链接工具生成

决策：新增二维码路径/链接生成工具，统一输出公开 H5 路径和绝对地址；后台页新增“打印二维码”按钮，触发海报预览与浏览器打印。

原因：二维码内容不依赖单条领料记录，而是指向固定公开入口，因此应该抽成独立链接工具，避免把二维码逻辑塞进列表行或选中项操作。海报与打印逻辑独立后，也便于后续复用到提示牌、桌贴等场景。

备选方案：

- 在列表中按行生成二维码：不符合业务目标，因为扫码应进入公共登记页而不是某条既有记录。
- 后台路由单独渲染打印页：可行，但比直接从按钮打开打印视图多一次导航成本。

## Risks / Trade-offs

- [公开插入能力可能被滥用] → 通过最小 RLS 范围控制，只开放种类读取和单表插入；继续保留数据库字段校验。后续如现场滥用明显，再补验证码或签名入口。
- [微信/手机视口下表单键盘遮挡] → 采用移动端单列布局与底部固定主按钮，并为页面底部预留安全区与滚动空间。
- [二维码指向错误环境或相对路径] → 统一由链接工具基于 `window.location.origin` 生成绝对地址，打印和页面展示都复用同一来源。
- [匿名策略影响现有后台权限模型] → 只新增 `select labor_protection_data` 与 `insert labor_protection_requisitions` 的公开策略，不修改后台管理员全权限策略。

## Migration Plan

1. 新增 Supabase migration，开放劳保种类公开读取和领料单公开插入。
2. 新增公开 H5 路由、页面组件、提交 hooks 和二维码链接工具。
3. 在后台劳保领料单页面增加二维码打印按钮与海报打印逻辑。
4. 通过构建验证前端改动，通过 migration 文件校验数据库改动语法与作用范围。

回滚策略：

- 前端回滚到变更前版本即可移除公开路由与打印入口。
- 数据库回滚时删除新增的公开 policy 即可恢复为管理员专用访问模式。

## Open Questions

- 当前不增加验证码或限流。如果现场部署后出现误填/刷单，需要另起变更补公开提交保护。
- 当前海报按单一固定文案输出，不支持多语言或多模板切换；如后续需要多区域张贴模板，可在打印能力上再扩展。
