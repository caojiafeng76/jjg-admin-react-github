## Context

多人录入被存储为员工级明细，列表按明细分页且仅能单条更新。导出依赖员工级明细，日报目前以 `created_at` 推断录入批次。数据库已对该表启用按页面权限的 RLS。

## Goals / Non-Goals

**Goals:**

- 持久化录入批次并以批次为列表与编辑单位。
- 让批次保存以单一数据库事务替换全部员工明细。
- 保持导出明细、工时与日报的既有员工核算口径。

**Non-Goals:**

- 不修改权限点、路由或导出的工作表结构。
- 不自动合并共同字段不一致的历史批次。

## Decisions

- 每条明细保存 `input_batch_id`；历史记录按相同 `created_at` 回填。这比仅依赖时间戳可靠，能在编辑后保持同一批次身份。
- 使用 `security invoker` RPC 保存批次。函数锁定原批次、删除旧明细并插入按员工均分的新明细，保证员工变更不会产生半完成数据；调用者继续受现有 RLS 约束。
- 列表由数据库按批次聚合并分页，导出继续直接读取明细。共同字段冲突的历史批次保留单条批次并返回冲突标记，由页面保留拆分展示。
- 日报分组键优先使用 `input_batch_id`，仅为未迁移数据回退到 `created_at`。

## Risks / Trade-offs

- [历史数据已有逐条修改] → 共同字段不一致的批次不自动合并，避免隐藏数据。
- [均分的一位小数会有总数舍入差] → 保持现有均分规则，避免改变已使用的工资和报表口径。
- [批次保存 RPC 需要生成类型] → 迁移应用后立即重新生成 `database.types.ts` 并编译验证。

## Migration Plan

1. 添加、回填并约束批次列，建立索引和保存/列表 RPC。
2. 生成数据库类型，部署服务与页面改动。
3. 验证历史批次计数、冲突批次保留拆分、批次保存及导出回归。
